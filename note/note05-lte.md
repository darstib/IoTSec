# LTE 蜂窝网络安全

## 蜂窝网络安全

### LTE（Long Term Evolution）概述 (page 2)

> [!definition]+ 什么是 4G LTE
>
> LTE 是第四代蜂窝技术标准，由第三代合作伙伴计划（3GPP）制定。它是从 GSM 到 UMTS 的演进步骤。

**LTE 的核心特性**：

1. **全球部署**：在全球范围内部署，安装数量快速增长
2. **全分组交换**：LTE 是完全基于分组交换的网络
3. **高数据速率**：提供更高的数据传输速率

### LTE 网络架构 (page 3)

LTE 网络主要由以下组件构成：

- **UE（User Equipment）**：用户设备，如智能手机
- **eNodeB（Evolved Node B）**：演进型基站，负责无线资源管理
- **MME（Mobility Management Entity）**：移动管理实体，负责移动性管理和认证
- **S-GW（Serving Gateway）**：服务网关，数据包路由
- **P-GW（PDN Gateway）**：分组数据网网关，连接到外部网络

![LTE 网络架构示意图|600](assets/note05-lte/w5p1-p03-lte-architecture.png)

从图中可以看到，UE 通过空中接口（Air Interface）连接到 eNodeB，eNodeB 再通过 S1 接口连接到核心网的 MME 和 S-GW。S-GW 与 P-GW 相连，P-GW 最终连接到外部的分组数据网（PDN），如互联网。这种分层架构使得 LTE 能够高效地处理数据流量，同时实现移动性管理和安全认证。

### LTE 网络协议栈 (page 4)

空中接口（UE 与 eNodeB 之间的无线链路）的通信协议栈分为三层：

**第三层（Layer 3）**：
- **RRC（Radio Resource Control）**：无线资源控制

**第二层（Layer 2）**：
- **PDCP（Packet Data Convergence Protocol）**：分组数据汇聚协议
- **RLC（Radio Link Control）**：无线链路控制
- **MAC（Medium Access Control）**：介质访问控制

**第一层（Layer 1）**：
- **PHY（Physical Access）**：物理接入层

![LTE 空中接口协议栈|600](assets/note05-lte/w5p1-p04-lte-protocol-stack.png)

协议栈的层次结构反映了不同的功能职责：物理层（PHY）负责实际的无线信号传输；MAC 层进行时频资源调度；RLC 层提供可靠传输和分段重组；PDCP 层处理数据压缩和加密；RRC 层则管理无线资源配置和移动性。理解这个协议栈对于分析 LTE 安全问题至关重要。

### LTE 帧结构 (page 5)

LTE 采用时频二维资源分配：

**时域结构**：
- 1 帧（Frame）= 10 ms
- 1 帧 = 10 子帧（Subframe）
- **1 子帧 = 1 ms**
- 1 子帧 = 2 时隙（Slot）
- **1 时隙 = 0.5 ms**
- 1 时隙 = 7 OFDM 符号（Symbols）

因此：1 帧 = 10 × 2 × 7 = 140 OFDM 符号

**频域结构**：
- 子载波间隔：15 kHz
- 资源块（Resource Block）：12 个子载波 × 7 OFDM 符号

![LTE 时频资源结构|600](assets/note05-lte/w5p1-p05-lte-frame-structure.png)

从图中可以看到，LTE 的时频资源呈现网格状结构。横轴是时间维度，纵轴是频率维度。最小的资源单元称为资源元素（Resource Element），由一个 OFDM 符号和一个子载波组成。资源块（Resource Block）是调度的基本单位，包含 12 个子载波和 7 个 OFDM 符号（即一个时隙）。这种精细的时频划分是信号遮蔽攻击能够精确对齐目标子帧的基础。



## LTE 安全攻击

### 信号遮蔽攻击（Signal Overshadowing Attack）(page 6)

#### 信号捕捉效应（Capture Effect）

当空中同时存在多个无线信号时，功率**最强**的那个信号会被接收方解调出来。

#### 信号遮蔽攻击（SigOver Attack）

**攻击原理**：
- 攻击者发送信号对目标子帧进行精准遮蔽（时域和频域上对齐）
- 利用信号捕捉效应，用高功率的攻击信号覆盖合法信号

**攻击效果**：

1. **若攻击信号无实际意义**：形成干扰（Jamming）
2. **若攻击信号是攻击者伪造的子帧**：攻击者能够向受害者注入伪造的广播信息

![SigOver 攻击原理示意图|600](assets/note05-lte/w5p1-p06-sigover-concept.png)

图中展示了信号遮蔽攻击的基本原理：合法基站发送的信号（蓝色）和攻击者发送的信号（红色）在空中重叠。由于攻击者离 UE 更近，其信号在 UE 处的功率更强，因此 UE 会解调出攻击者的信号而忽略合法基站的信号。关键在于，攻击者必须在时域和频域上精确对齐目标子帧，才能成功遮蔽。

### 攻击模型 (page 7)

**攻击者的能力**：
- 攻击者**不需要**知道受害者 UE 的密钥
- 受害者与基站之间的加密传输无法被解密
- 攻击者能够**监听**合法基站到受害者 UE 的广播信息

**为什么可以攻击？**

LTE 中的广播信息（如系统信息块 SIB）是**未加密**的，所有 UE 都能接收。攻击者可以监听、修改并重新发送这些广播信息。

![SigOver 攻击模型图|600](assets/note05-lte/w5p1-p07-sigover-attack-model.png)

该图展示了完整的攻击场景：攻击者位于合法基站和目标 UE 之间。攻击者首先监听并捕获基站发送的广播信息（步骤 1），然后修改并构造恶意子帧（步骤 2），最后在精确的时间点以更强的功率发送恶意子帧（步骤 3），遮蔽基站的合法信号。由于 UE 仍然与基站保持连接，它不会意识到自己接收到的是伪造信息。

### 构造恶意子帧 (page 8)

**攻击步骤**：

1. **通信配置的匹配**
   - 识别受害者所接入合法基站的物理配置信息
   - 通过监听可完成

2. **构造符合 LTE 标准的子帧**
   - 恶意子帧必须符合 LTE 标准
   - 由多种 LTE 信道组成（PCFICH、PDCCH、PDSCH）

3. **包含导频信号（Pilot Signal）**
   - 通过包含导频信号来帮助受害者成功解调出恶意子帧中的信息
   - 导频信号用于信道估计和同步

![构造恶意子帧的组成|600](assets/note05-lte/w5p1-p08-sigover-malicious-frame.png)

从图中可以看到，一个完整的 LTE 子帧包含多个信道：PCFICH（物理控制格式指示信道）告诉 UE 控制区域的大小；PDCCH（物理下行控制信道）携带调度信息；PDSCH（物理下行共享信道）承载实际的用户数据和广播信息。攻击者必须构造一个符合标准的子帧，包含所有必要的信道和导频信号，才能让 UE 成功解调。图中的网格结构展示了导频信号（红色标记）在时频资源中的分布，这些导频对于 UE 进行信道估计至关重要。

### 精确的信号遮蔽：时域同步 (page 9-10)

**时域同步的必要性**：

LTE 的子帧有严格的时间顺序，攻击信号必须与合法信号在时域上对齐才能成功遮蔽。

**时域同步方法**：
- 使用合法基站的同步信号（PSS/SSS，Primary/Secondary Synchronization Signal）完成时域上的同步
- 确定合法基站的帧/子帧的时序

**传输时延问题**：

攻击者发送的信号存在时延，它随着攻击者、基站、受害者三者位置的不同而改变。

**解决方案**：
- 手机的基带处理器本身有一定的延迟容忍度
- 对于一个典型的覆盖半径为 1.5 km 的基站而言，攻击者最大时延为 10 μs
- 这个时延在手机所能容忍的范围内

### 精确的信号遮蔽：频域同步 (page 11)

**频率精度要求**：

LTE 标准规定了基站的最小频率精度：
- 十亿分之 50 的偏差（50 ppb, parts per billion）
- 在 1.8 GHz 频段：±90 Hz（计算：$1.8 \times 10^9 \times 50 \times 10^{-9} = 90$ Hz）

**攻击者的解决方案**：

使用 **GPS 定向振荡器（GPSDO, GPS Disciplined Oscillator）**：
- 十亿分之 25 的偏差（25 ppb）
- 在 1.8 GHz 频段：±45 Hz（计算：$1.8 \times 10^9 \times 25 \times 10^{-9} = 45$ Hz）
- 比标准要求更高的频率精度
- 确保攻击信号与合法信号在频域上精确对齐

### 伪基站（FBS）vs 信号遮蔽（SigOver）(page 12)

**相同点**：
- FBS 和 SigOver 都能够向 UE 发送广播信息

**SigOver 的优势**：

1. **功率优势**
   - 不需要比合法基站功率更大，只需要在受害者 UE 处功率更强
   - 攻击者可以靠近受害者，而合法基站距离较远

2. **不需要与基站建立连接**
   - 伪基站需要伪装成合法基站，诱导 UE 连接
   - SigOver 只是遮蔽信号，无需建立连接

3. **受害者 UE 与合法基站保持通信**
   - UE 不会意识到被攻击
   - 正常的通信服务不受影响（除了被注入的伪造信息）

![SigOver 与伪基站对比|600](assets/note05-lte/w5p1-p12-sigover-vs-fbs.png)

图中清晰对比了两种攻击方式的差异。左侧的伪基站（FBS）攻击需要诱导 UE 断开与合法基站的连接并重新连接到伪基站，这要求伪基站具有足够强的信号覆盖，并且 UE 会检测到网络切换。右侧的 SigOver 攻击则让 UE 保持与合法基站的连接，只是在特定子帧上用攻击者的信号遮蔽合法信号，因此更加隐蔽且不易被察觉。

### 攻击鲁棒性测试 (page 13)

**测试环境**：

- **(A) LOS（Line of Sight，视距）**：距离 2m
- **(B) NLOS（Non-Line of Sight，非视距）**：距离 10m

**实验结果**：

SigOver 攻击的成功率显著高于伪基站（FBS）攻击：
- **功率优势**：SigOver 在较低功率下即可成功
- **鲁棒性**：SigOver 在 NLOS 环境下仍能保持高成功率

![SigOver 攻击鲁棒性测试结果|600](assets/note05-lte/w5p1-p13-sigover-robustness.png)

图中展示了实验测试结果，横轴是攻击者的发射功率，纵轴是攻击成功率。可以看到，SigOver 攻击（实线）在两种环境下都表现出色：在 LOS 环境（蓝色实线）下，即使功率较低也能达到近 100% 的成功率；在 NLOS 环境（红色实线）下，成功率略有下降但仍然很高。相比之下，传统的伪基站攻击（虚线）需要更高的功率才能达到相同的成功率，尤其在 NLOS 环境下表现明显较差。这证明了 SigOver 攻击的实用性和威胁性。

### SigOver 攻击应用：信令风暴 (page 14-16)

#### 常规的信令风暴攻击

**传统方法**：
- 需要大量的恶意设备发起 DDoS 攻击
- 成本高、易被检测

#### SigOver 实现信令风暴

**攻击机制**：
- 通过伪造的子帧触使受害者 UE 与基站进行频繁交互
- 可以对一个基站下**所有 UE** 进行攻击

**攻击效率** (page 16)：

| 场景 | 位置更新请求（每小时） | 所有信令（每小时） | 倍数 |
||-|--||
| **正常使用** | 45 次 | 675 次 | - |
| **SigOver 攻击** | 21,600 次 | 432,000 次 | **640 倍** |

> [!attention]+ 攻击威力
>
> 仅使用一台设备，SigOver 就能让单个 UE 每小时产生 432,000 次信令，相当于正常使用的 640 倍。如果同时攻击多个 UE，可以迅速导致基站信令系统过载。

### 防御措施 (page 17-18)

#### 方法一：利用信道的不同 (page 17)

**思路**：
- 检测信号的物理层特征差异，区分合法信号和攻击信号
- 例如：信道响应、接收功率等

**问题**：
- 鲁棒性还需提升
- NLOS（非视距）环境下难以实现区分

#### 方法二：对广播信息进行数字签名 (page 18)

**思路**：
- 基站对广播信息进行数字签名
- UE 验证签名，拒绝未签名或签名无效的广播信息

**部署挑战**：

1. **全球公钥体制**
   - 对于全球漫游场景，需要一个全球统一的公钥基础设施（PKI）
   - 管理证书需要额外开销

2. **技术挑战**
   - 广播信息周期很短（如 40/80 ms）
   - 认证证书和签名会导致 UE 设备更高的电量开销
   - 签名验证增加 UE 的计算负担



## VoLTE 安全攻击

### VoLTE 概述 (page 19)

**VoLTE（Voice over LTE）**：通过 4G 流量进行语音通话。

**全球部署情况**：
- 全球有 253 家运营商在使用 VoLTE
- 手机上的"VoLTE"图标或"HD"图标表示正在使用 VoLTE

**VoLTE 的优势**：

1. **核心网处理时间更快**
2. **更高清高质量的通话**
3. **对于运营商而言使用 VoLTE 更有效率**
   - 统一使用分组交换网络，无需维护独立的电路交换语音网络

### VoLTE 架构 (page 20)

**核心组件**：

- **IMS（IP Multimedia Subsystem，IP 多媒体子系统）**
  - UE 与 IMS 建立联系
  - IMS 管理 VoLTE 的呼入/呼出

**工作流程**：
1. UE 通过 LTE 数据连接到 IMS
2. IMS 处理呼叫控制（SIP 协议）
3. 语音数据通过 RTP（Real-time Transport Protocol）传输

![VoLTE 网络架构|600](assets/note05-lte/w5p2-p20-volte-architecture.png)

从图中可以看到，VoLTE 架构在 LTE 网络的基础上增加了 IMS 核心网组件。UE 通过 LTE 的 EPC（演进分组核心网）连接到 IMS，IMS 包含多个功能实体：P-CSCF（代理呼叫会话控制功能）、S-CSCF（服务呼叫会话控制功能）和 I-CSCF（询问呼叫会话控制功能）。这些组件共同处理 SIP 信令，实现呼叫的建立、维护和释放。语音数据本身则通过 RTP 协议在 UE 之间传输，经过 EPC 的承载通道。这种全 IP 架构使得 VoLTE 能够提供高质量的语音服务，同时简化了网络设计。

### VoLTE 的承载（Bearer）(page 21)

> [!definition]+ 承载（Bearer）
>
> 承载是指从核心网到 UE 的**逻辑通路**。每个承载有唯一的 Bearer ID，用于承载特定类型的数据流。

**VoLTE 语音数据承载**：
- 包含语音数据的 RTP 流
- 使用专用的 Bearer ID
- 与普通数据流分离

### LTE 的流加密机制 (page 22)

LTE 使用流加密（Stream Cipher）保护数据传输。加密密钥流的生成依赖以下参数：

**密钥流生成参数**：

1. **Key (k_up)**：用户平面密钥
   - 用于 VoLTE 的密钥
   - 每次建立无线电连接时会产生一个新的密钥

2. **Count**：数据包的序列号
   - 用于防止重放攻击
   - 每发送一个数据包，Count 递增

3. **Bearer**：当前加密所用于的承载 ID

4. **Direction**：上行或下行

5. **Length**：流密钥块的长度
   - 不会影响密钥生成的结果

**加密算法**：
- **EEA（EPS Encryption Algorithm）**：EPS 加密算法
- 常用的有 EEA1（SNOW 3G）、EEA2（AES-CTR）、EEA3（ZUC）

**流加密过程**：

密钥流生成函数：
$$\text{Keystream} = \text{EEA}(Key, Count, Bearer, Direction, Length)$$

加密过程：
$$\text{Ciphertext} = \text{Plaintext} \oplus \text{Keystream}$$

其中 EEA 是加密算法（如 EEA1/EEA2/EEA3）。

![LTE 流加密机制示意图|575](assets/note05-lte/w5p2-p22-lte-stream-cipher.png)

图中详细展示了 LTE 流加密的工作流程。加密算法（如 SNOW 3G、AES 或 ZUC）接收五个输入参数：密钥 Key、计数器 Count、承载 ID Bearer、方向 Direction 和长度 Length，生成与明文等长的密钥流 Keystream。明文数据与密钥流进行异或（XOR）运算得到密文。这个过程的关键在于，相同的五个参数会产生完全相同的密钥流。因此，如果任何一个参数重复使用，就会导致密钥流重用，进而产生严重的安全漏洞。ReVoLTE 攻击正是利用了这一点。

### 密钥重用问题 (page 23)

**ReVoLTE 攻击原理**：重用加密的 VoLTE 流量来监听通话。

**为什么密钥重用会导致密文被解密？**

对于流加密，如果同一密钥流 $K$ 被用于加密两段不同的明文 $M_1$ 和 $M_2$：

$$C_1 = M_1 \oplus K$$
$$C_2 = M_2 \oplus K$$

攻击者如果知道 $M_2$ 和 $C_1$、$C_2$，就可以推导出 $M_1$：

$$C_1 \oplus C_2 = M_1 \oplus M_2$$
$$M_1 = C_1 \oplus C_2 \oplus M_2$$

### VoLTE 中的密钥重用场景 (page 24-27)

#### 第一次通话 (page 24-25)

1. **RRC Security Mode Command**：建立加密链路，生成密钥 Key (k_up)
2. **First Call**：用户拨打第一个电话
   - VoLTE **重置 COUNT**
   - **设置 BEARER ID**

![第一次通话建立过程|600](assets/note05-lte/w5p2-p24-volte-key-reuse-1.png)

上图展示了第一次通话的初始化阶段。UE 与网络建立 RRC 连接后，通过 Security Mode Command 协商加密算法并生成会话密钥 k_up。这个密钥在整个 RRC 连接期间保持不变。

![第一次通话的密钥参数设置|600](assets/note05-lte/w5p2-p25-volte-key-reuse-2.png)

当建立第一个 VoLTE 呼叫时，系统为语音流量分配一个专用的 Bearer（如 Bearer ID = 5），并将 PDCP COUNT 重置为 0 开始计数。此时，加密密钥流由五元组 (Key, Count=0, Bearer=5, Direction, Length) 确定。

#### 第二次通话 (page 26-27)

3. **Second Call**：用户拨打第二个电话
   - VoLTE **再次重置 COUNT**
   - **设置 BEARER ID**

![第二次通话建立|600](assets/note05-lte/w5p2-p26-volte-key-reuse-3.png)

第一次通话结束后，如果 RRC 连接未断开，密钥 k_up 仍然有效。当用户发起第二次通话时，系统会再次分配 Bearer 并重置 COUNT。

![密钥重用漏洞产生|600](assets/note05-lte/w5p2-p27-volte-key-reuse-4.png)

**关键问题**：如果两次通话使用相同的 BEARER ID，则会出现密钥重用！

因为：
- Key (k_up) 相同（无线电连接未断开）
- Count 被重置为相同的初始值
- Bearer ID 相同
- Direction 相同

这意味着两次通话使用了**完全相同的密钥流**。图中清晰展示了这个问题：第一次通话的数据包 0、1、2... 和第二次通话的数据包 0、1、2... 使用了相同的密钥流进行加密。攻击者可以利用这一点，通过已知的第二次通话明文来推导第一次通话的明文。

### 基站的 Bearer ID 分配缺陷 (page 28)

**不安全的基站**：

测试发现，部分基站存在 Bearer ID 重用问题：
- **3 / 15 基站**：每次都增加 Bearer ID 的值（安全）
- **12 / 15 基站**：可能重用 Bearer ID（**存在密钥重用漏洞**）

**漏洞根本原因**：VoLTE 规范对 Bearer ID 的分配策略描述模棱两可，没有明确禁止重用。

### ReVoLTE 攻击流程 (page 29-30)

**攻击场景**：Alice 和 Bob 通话，攻击者想监听通话内容。

#### 第一步：目标通话（第一次通话）(page 29)

- **Bob** 和 **Alice** 进行通话
- **攻击者**通过嗅探器（Sniffer）捕获加密的语音流量
- 记录下密文 $C$（目标通话的加密数据）

![ReVoLTE 攻击第一步：捕获目标通话|600](assets/note05-lte/w5p2-p29-revolte-attack-1.png)

图中展示了攻击的第一阶段。Bob 和 Alice 正在进行 VoLTE 通话，语音数据经过 LTE 加密后在空中传输。攻击者使用软件定义无线电（如 USRP）被动嗅探下行链路，捕获所有加密的数据包并保存。此时攻击者只有密文 $C$，无法直接解密，因为不知道密钥流。但攻击者会记录下这些密文，等待下一步操作。

#### 第二步：密钥通话（第二次通话）(page 30)

- **攻击者**在发现上一个通话结束后，**立即**向 Alice 发起第二次通话
- 攻击者收集通话过程中的：
  - 明文 $M'$（攻击者知道自己说了什么）
  - 密文 $C'$（嗅探获得）

![ReVoLTE 攻击第二步：发起密钥通话|600](assets/note05-lte/w5p2-p30-revolte-attack-2.png)

第一次通话结束后，攻击者立即向 Alice 拨打电话。关键时机是在 RRC 连接尚未释放之前，这样密钥 k_up 仍然有效。在第二次通话中，攻击者可以播放预先录制的音频或说出已知内容，因此完全知道明文 $M'$。同时，攻击者继续嗅探空中接口，捕获这次通话的密文 $C'$。如果基站重用了相同的 Bearer ID，那么两次通话的密钥流完全相同。

#### 第三步：解密目标通话

结合第一步的密文 $C$ 和第二步的明文 $M'$、密文 $C'$，即可推导出目标通话的明文 $M$：

$$M = C \oplus C' \oplus M'$$

**为什么能解密？**

因为两次通话使用了相同的密钥流 $K$：
- $C = M \oplus K$
- $C' = M' \oplus K$
- $C \oplus C' = M \oplus M'$
- $M = C \oplus C' \oplus M'$

攻击者利用已知的 $M'$ 作为"密钥"，通过简单的异或运算即可恢复出 Bob 和 Alice 第一次通话的内容。这种攻击不需要破解加密算法，也不需要获取密钥，完全是利用了实现层面的密钥重用漏洞。

### 实际实验结果 (page 31)

**实验设备**：
- 3 台 UE（用户设备）
- 1 台连接了 USRP B210 的笔记本
  - 用于下行链路的嗅探和攻击的实现

**实验结果**：
- **解密了 89% 的数据包**

**信息损失的原因**：
- 下行链路的嗅探没能捕捉到所有的无线数据包
- 在目标通话中损失了 3%
- 在密钥通话中损失了 8%

### 漏洞分析 (page 32)

**总结**：

在 UE 与基站保持连接的情况下连续进行两次通话时：
- COUNT 会被重置
- Bearer ID 可能被重用
- 导致最终密钥流的重用

**实现上的缺陷**：

VoLTE 的规范模棱两可：
- 禁止重复使用密钥流
- 但并没有指定一个确切的实现方式
- 没有明确要求每次通话必须使用不同的 Bearer ID

> [!note]+ 规范的模糊性
>
> 标准文档通常规定"不应该"（should not）重用密钥流，但没有强制要求"必须不"（must not）使用某种特定的实现方式。这给实现者留下了解释空间，导致不同厂商的基站行为不一致。

### 防御措施 (page 33)

#### 方法一：在不同通话中使用不同的 Bearer ID

**实施方式**：
- 基站在分配 Bearer ID 时，确保每次通话使用不同的 ID
- 即使 COUNT 被重置，由于 Bearer ID 不同，密钥流也会不同

**优点**：
- 实现简单
- 无需修改协议标准
- 对 UE 无额外负担

**缺点**：
- Bearer ID 数量有限（VoLTE 可用的 EPS Bearer ID 范围为 5-15，共 11 个可用 ID）
- 需要管理 Bearer ID 的分配和回收

#### 方法二：加密 RTP 流量

**实施方式**：
- 在应用层对 RTP 流量进行端到端加密
- 使用 SRTP（Secure Real-time Transport Protocol）

**优点**：
- 即使 LTE 层密钥重用，应用层加密仍然有效
- 端到端安全，不依赖于网络层的安全性

**缺点**：
- 需要 UE 和 IMS 都支持 SRTP
- 增加计算和电量开销
- 需要密钥协商机制（如 DTLS-SRTP）



## 总结

本周内容涵盖了无线网络的介质访问控制（MAC）协议和蜂窝网络（LTE）的安全问题。

### MAC 协议

- **信道分割**：TDMA、FDMA、CDMA，适合高负载场景，但低负载时效率低
- **随机访问**：ALOHA、CSMA、CSMA/CD、CSMA/CA，适合动态负载，但存在碰撞问题
- **轮流**：轮询、令牌传递，公平有序，但存在单点故障风险

### LTE 安全攻击

1. **信号遮蔽攻击（SigOver）**
   - 利用信号捕捉效应，用高功率信号遮蔽合法信号
   - 可注入伪造的广播信息
   - 可发起信令风暴攻击，导致基站过载

2. **ReVoLTE 攻击**
   - 利用密钥流重用漏洞，解密 VoLTE 通话
   - 根本原因：COUNT 重置 + Bearer ID 重用
   - 攻击成功率高达 89%

**防御思路**：
- 对广播信息进行数字签名（部署难度大）
- 避免 Bearer ID 重用
- 使用应用层加密（SRTP）
