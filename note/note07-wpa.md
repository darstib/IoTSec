# Wi-Fi Protected Access (WPA)

## WPA 的诞生背景 (page 12)

**Wi-Fi Protected Access (WPA)** 是无线局域网（WLAN）行业对 WEP 协议严重安全缺陷的快速响应，旨在提供比 WEP 更强的即时安全解决方案。

WPA 设计目标：
- **软件/固件升级**：可以通过软件或固件升级现有接入点，无需更换硬件
- **低成本实施**：实施时间短、成本低
- **厂商兼容性**：兼容不同厂商的设备
- **广泛适用性**：适合企业、小型站点和家庭网络

> [!note] WPA 的市场定位
> 
> WPA 是一个过渡性解决方案，在 WEP 被证明不安全后，工业界需要一个能够快速部署、向后兼容现有硬件的临时方案。WPA 的核心思想是在不更换硬件的前提下，通过软件升级尽可能地修补 WEP 的漏洞。

---

## WPA 如何解决 WEP 的漏洞 (page 23)

WPA 通过在 RC4 密码引擎外围包裹四个新算法来解决 WEP 的安全问题：

### 1. 消息完整性代码（MIC）：Michael 算法

- **设计目标**：为已部署的硬件设计，必须能在旧设备上运行
- **主动对抗措施**：需要配合主动防御机制使用

### 2. 扩展的 48 位 IV 与 IV 序列规则

- **IV 空间扩展**：48 位 IV 提供 $2^{48}$ 种可能性，超过 500 万亿
- **序列规则**：明确规定 IV 的选择和验证方式，防止重放攻击

### 3. 密钥派生与分发

- **密钥层次结构**：从主密钥（Master Key）派生出密钥层次体系
- **密钥刷新**：主密钥可以安全地刷新更新

### 4. 临时密钥完整性协议（TKIP）

- **逐包密钥生成**：为每个数据包生成独立的加密密钥

## 临时密钥完整性协议（TKIP）

### TKIP 的逐包密钥混合机制 (page 3)

**WEP 与 TKIP 的密钥构造对比**：

| 协议 | 密钥构造方式 |
|------|--------------|
| **WEP** | 简单拼接基础密钥和 IV：`Per-packet Key = Base Key \|\| IV` |
| **TKIP** | 经过两阶段密钥混合：Phase 1 + Phase 2 |

**TKIP 密钥混合流程**：

```
Temporal Key (时间密钥)  ─┐
MAC Address (MAC 地址)   ─┤→ Phase 1 → Intermediate Key (中间密钥) ─┐
                          │                                        │
Sequence Number (序列号) ─┴────────────────────────────────────────┤→ Phase 2 → Per-packet Key
```

> [!note] 设计动机
> 
> 两阶段混合机制的目的是让攻击者**难以关联 IV 和逐包密钥**。即使攻击者捕获了大量数据包和对应的 IV，也无法轻易推导出密钥生成规律，从而有效缓解了 WEP 中的弱密钥问题。

### TKIP 加密与解密流程 (page 4-5)

TKIP 的加密和解密过程仍然使用 **RC4 流密码**，但在密钥生成和完整性保护方面进行了增强。具体流程在课件中以图示形式展现，核心改进包括：
- 使用两阶段混合生成的逐包密钥
- 加入 Michael MIC 进行完整性校验
- 使用扩展的 IV 序列号

### TKIP 总结与评价 (page 6)

**外部专家评审**：Ron Rivest、David Wagner、John Kelsey、Susan Langford 等密码学专家对 TKIP 进行了评审。

**TKIP 的优势**：
- ✓ 提供**弱但真实的安全性**（weak but genuine security）
- ✓ 实现了在**几乎所有现有设备**上通过软件部署的目标
- ✓ 相比 WEP **性能下降不明显**
- ✓ 满足市场对基于现有设备基础的迁移路径需求

**TKIP 的定位**：过渡性解决方案，不是长期安全标准。

## WPA 的两种工作模式

### 模式对比 (page 7)

| 特性 | Pre-Shared Key Mode（预共享密钥模式） | Enterprise Mode（企业模式） |
|------|--------------------------------------|----------------------------|
| **适用场景** | 家庭 / 小型办公（SOHO） | 企业、政府、教育机构 |
| **认证服务器** | 不需要 RADIUS 服务器 | 需要认证服务器（RADIUS） |
| **密钥管理** | AP 本地管理 | 使用 802.1X 集中管理用户凭证 |
| **认证方式** | 使用共享密钥（Shared Secret）认证到 AP | 使用 RADIUS 协议进行认证和密钥分发 |
| **主要威胁** | 易受字典攻击 | 目前未发现有效攻击 |

## WPA-PSK（预共享密钥模式）详解

### WPA-PSK 概述 (page 8)

**适用场景**：家庭和小型办公（SOHO，Small Office/Home Office）

**核心机制**：
1. **四次握手认证**（4-Way Handshake Authentication）
   - 进行身份认证
   - 生成会话密钥 PTK（Pairwise Transient Key）

2. **预共享密钥 + TKIP**
   - 无标准化的共享密钥更换方式
   - 如果共享密钥泄露，网络安全可能受损

**安全特性**：
- ✓ 显著提高被动监听和解密流量的难度
- ✗ 易受**被动字典攻击**（passive dictionary attack）
- **密钥复杂度越高，越难受字典攻击**

### 密钥层次结构

#### 成对主密钥（PMK）的构造 (page 36, page 2)

**PMK 生成公式**：

$$
\text{PMK} = \text{PBKDF2}(\text{Passphrase}, \text{SSID}, \text{SSID Length}, 4096, 256)
$$

**参数说明**：
- **Passphrase**：用户输入的密码短语
- **SSID**：无线网络名称，作为盐值（salt）
- **SSID Length**：SSID 的长度
- **迭代次数**：4096 次，增加暴力破解难度
- **输出长度**：256 位（32 字节）

> [!note] PBKDF2 函数
> 
> **PBKDF2**（Password-Based Key Derivation Function 2）是一种基于口令的密钥派生函数。通过多次迭代哈希运算，将用户密码和盐值转换为固定长度的密钥。
> 
> **Why SSID as salt?** 使用 SSID 作为盐值意味着相同的密码在不同的网络中会生成不同的 PMK，防止预计算攻击（如彩虹表）在多个网络间复用。

#### 成对瞬时密钥（PTK）的构造 (page 2)

**PTK 生成公式**：

$$
\text{PTK} = \text{HASH}(\text{PMK}, \text{ANonce}, \text{SNonce}, \text{AP MAC}, \text{STA MAC})
$$

**输入参数**：
- **PMK**：成对主密钥（从密码派生或 802.1X 协商获得）
- **ANonce**：AP（Authenticator）生成的随机数
- **SNonce**：STA（Station，客户端）生成的随机数
- **AP MAC Address**：接入点的 MAC 地址
- **STA MAC Address**：客户端的 MAC 地址

**设计思想**：
- 引入**随机数（Nonce）**，确保每次连接生成的 PTK 都不同
- 绑定**双方 MAC 地址**，防止密钥在不同设备间重放
- PTK 是**会话密钥**，断开连接后失效，重新连接时重新生成

### 四次握手协议（4-Way Handshake）(page 9-10)

**四次握手的双重作用**：
1. **相互认证**：验证双方都持有正确的 PMK
2. **密钥协商**：生成并确认 PTK

**握手流程简述**：
1. **Message 1**：AP → STA，发送 ANonce
2. **Message 2**：STA → AP，发送 SNonce + MIC（使用 PTK 计算）
3. **Message 3**：AP → STA，确认 PTK，发送 GTK（Group Temporal Key）
4. **Message 4**：STA → AP，确认收到

> [!example] 四次握手的安全依赖
> 
> 四次握手受 PTK 保护，而 **PTK 的强度依赖于密码的强度**。如果密码弱，整个认证过程的安全性就会崩塌。

### WPA-PSK 的漏洞与攻击 (page 11)

**核心漏洞**：
- PSK 公式为 `PSK = PBKDF2(Password, SSID, SSIDlength, 4096, 256)`
- **四次握手的强度完全依赖密码强度**

**可行的攻击方式**：

#### 1. 离线暴力破解和字典攻击

**攻击步骤**：
1. **捕获四次握手消息**：攻击者被动监听网络，记录完整的四次握手过程
2. **离线破解**：在本地对捕获的握手消息进行暴力破解或字典攻击

**为什么离线攻击可行？**
- 握手消息包含足够信息验证密码猜测是否正确
- 攻击者无需与 AP 交互，可以无限次尝试
- 不受网络速率限制，破解速度仅取决于计算资源

### WPA-PSK 暴力破解的时间成本 (page 13-14)

**假设条件**：
- 密码字符集：`[A-Z][a-z][0-9]`，共 62 种字符
- 最小密码长度：8 个字符（最大 63 个）
- 硬件：NVIDIA GeForce 280GTX（11,000 keys/s）

**计算过程**：

$$
\begin{align}
\text{密钥空间} &= 62^8 = 218{,}340{,}105{,}584{,}896 \\
\text{破解时间} &= \frac{218{,}340{,}105{,}584{,}896}{11{,}000 \text{ keys/s}} \\
&= 19{,}849{,}100{,}508 \text{ 秒} \\
&= 5{,}513{,}639 \text{ 小时} \\
&= 229{,}735 \text{ 天} \\
&= \mathbf{630 \text{ 年}}
\end{align}
$$

**结论**：
- 8 字符纯字母数字密码：**630 年**
- 12 字符密码：**9,309,091,680 年**
- 对于简单密码短语，暴力破解耗时过长
- **字典攻击仍然是更好的解决方案**

### 彩虹表攻击（Rainbow Table Attack）(page 15)

**彩虹表原理**：
- **预计算索引哈希列表**：预先计算数百万个常用密码针对常见 SSID 的哈希值
- **存储需求**：2-3 TB 的存储空间
- **覆盖范围**：已知的彩虹表可覆盖最多 14 个字符的密码
- **适用版本**：WPA v1 和 v2（使用预共享密钥）

**攻击优势**：
- **现场验证速度极快**：无需实时计算，直接查表
- 大幅减少破解时间

**防御**：使用长且复杂的密码，避开常见密码列表。

### WPA-PSK 使用建议 (page 16)

**安全风险**：
- 如果密码短，易受外部字典攻击
- WPA-PSK 应当**谨慎使用**

**推荐方案**：
- **优先选择 WPA + 802.1X（企业模式）**
- 目前尚未发现针对 WPA 企业模式的有效攻击

## WPA 企业模式（WPA Enterprise Mode）

### 企业模式概述 (page 17)

**核心特性**：
- 认证服务器使用 **802.1X** 为每个用户分发不同的密钥
- 提供**增强的安全性和认证**
- **适用场景**：企业、政府、教育机构

**与 PSK 模式的本质区别**：
- PSK 模式：所有用户共享一个密钥
- 企业模式：每个用户拥有独立的凭证和密钥

### IEEE 802.1X 标准 (page 18)

**定义**：IEEE 802.1X 是一个针对局域网（LAN）的**基于端口的网络访问控制**（Port-Based Network Access Control）标准。

**在 802.11 架构中的作用**：
- 填补了 802.11 架构中**访问控制、认证和密钥管理**的安全漏洞
- 认证通常由**第三方实体**完成，例如 RADIUS（Remote Authentication Dial-In User Service）服务器

### 802.1X 的三个主要组件 (page 19)

| 组件 | 英文名称 | 角色 | 示例 |
|------|----------|------|------|
| **请求者** | Supplicant | 需要访问网络的客户端 | 无线客户端 |
| **认证器** | Authenticator | 网络接入控制点 | 接入点（AP） |
| **认证服务器** | Authentication Server | 验证用户身份 | RADIUS 服务器 |

**工作流程**：
1. Supplicant 请求访问网络
2. Authenticator（AP）充当中间人，转发认证消息
3. Authentication Server 验证用户凭证，决定是否授权

### 可扩展认证协议（EAP）(page 20-22)

#### EAP 的定义 (page 21)

**EAP**（Extensible Authentication Protocol）是一个**载体协议**（carrier protocol），设计用于传输"真实"认证协议的消息（如 TLS）。

**EAP 的特性**：
- **通用（灵活）认证协议**：可以运行在任何链路层协议之上
- **提供通用功能**：协商所需的认证机制（称为 methods）
- **基于 EAP 和关联阶段协商的具体认证方法** (page 20)

#### EAP 的四种消息类型 (page 22)

EAP 非常简单，只有四种消息类型：

1. **EAP Request**：从认证服务器（AS）到移动设备（M）的消息
2. **EAP Response**：从移动设备（M）到认证服务器（AS）的消息
3. **EAP Success**：表示认证成功
4. **EAP Failure**：表示认证失败

**重要特性**：
- **Authenticator（AP）不理解 EAP 消息内部内容**，只识别 EAP Success 和 Failure
- **EAP 本身不是认证方法**，而是承载认证方法的框架

### EAP 的封装协议 (page 24-25)

#### EAPoL（EAP over LAN）[802.1X] (page 25)

**定义**：将 EAP 消息封装到局域网协议中（例如以太网）

**用途**：在客户端和 AP 之间传输 EAP 消息

#### RADIUS（Remote Access Dial-In User Service）(page 25)

**用途**：在 AP 和认证服务器之间传输 EAP 消息

#### 协议栈示意 (page 24-25)

```
移动客户端 (Mobile Client)        接入点 (Access Point)        认证服务器 (Auth Server)
┌─────────────┐                   ┌─────────────┐               ┌─────────────┐
│   EAP-TLS   │                   │             │               │   EAP-TLS   │
├─────────────┤                   │             │               ├─────────────┤
│     EAP     │ ←─────────────→   │             │               │     EAP     │
├─────────────┤                   ├─────────────┤               ├─────────────┤
│ EAPoL       │ ←──── 802.11 ──→  │ EAPoL       │               │             │
│ (802.1X)    │                   │ (802.1X)    │               │             │
└─────────────┘                   ├─────────────┤               │             │
                                  │ EAP over    │ ←── RADIUS ──→│             │
                                  │   RADIUS    │               │             │
                                  ├─────────────┤               ├─────────────┤
                                  │   TCP/IP    │               │   TCP/IP    │
                                  ├─────────────┤               ├─────────────┤
                                  │   802.3     │               │   802.3     │
                                  └─────────────┘               └─────────────┘

     ↑                                   ↑                             ↑
     └─── IEEE 802.11i 范围内 ───────────┘                             │
                                                                       └─ 超出范围
```

**关键点**：
- 客户端到 AP：EAP 封装在 **EAPoL** 中，通过 **802.11** 无线传输
- AP 到认证服务器：AP 代理 EAP，使用 **RADIUS** 协议封装
- **端到端传输**：客户端和认证服务器之间 EAP 端到端传输
- AP 是透明的中继者，不解析 EAP 内容

### EAP 实际工作流程 (page 26)

```
客户端 (Client)          AP                  认证服务器 (Auth Server)
     │                   │                           │
     ├─ EAPoL-Start ────→│                           │
     │                   │                           │
     │←─ EAP Request ────┤                           │
     │  (Identity)       │                           │
     │                   │                           │
     ├─ EAP Response ───→│                           │
     │  (Identity)       ├─ EAP Response (Identity)→│
     │                   │                           │
     │                   │←─── EAP Request 1 ────────┤
     │←─ EAP Request 1 ──┤                           │
     │                   │                           │
     ├─ EAP Response 1 ─→│                           │
     │                   ├──── EAP Response 1 ───────→│
     │                   │                           │
     │       ...         │          ...              │
     │                   │                           │
     │                   │←─── EAP Request n ────────┤
     │←─ EAP Request n ──┤                           │
     │                   │                           │
     ├─ EAP Response n ─→│                           │
     │                   ├──── EAP Response n ───────→│
     │                   │                           │
     │                   │←──── EAP Success ─────────┤
     │←── EAP Success ───┤                           │
     │                   │                           │

 封装在 EAPoL 中                      封装在 RADIUS 中
   (802.11 传输)                         (有线网络传输)
```

**流程说明**：
1. 客户端发送 **EAPoL-Start** 启动认证
2. AP 请求身份标识（Identity）
3. 客户端响应身份，AP 转发给认证服务器
4. **嵌入的认证协议**（如 EAP-TLS）开始多轮请求/响应交换
5. 认证服务器发送 **EAP Success** 表示认证成功
6. AP 转发成功消息给客户端

### IEEE 802.1X 认证总结 (page 27)

**认证结束时的状态**：

1. **会话建立**：认证服务器和客户端已建立会话
2. **主密钥生成**：认证服务器和客户端持有相互认证的**主密钥**（Master Key，从具体的 EAP 方法派生）
3. **访问授权令牌**：主密钥代表基于认证的访问授权决定
4. **PMK 派生**：客户端和认证服务器已派生 **PMK**（Pairwise Master Key）
5. **PMK 分发**：认证服务器已将 PMK 分发给 AP
6. **强制访问控制**：PMK 是一个授权令牌，用于在 AP 处强制执行访问控制决定

> [!note] PMK 的来源
> 
> 在企业模式中，PMK 不是从密码派生，而是通过 802.1X 认证过程中的密钥交换协商生成。这使得每个用户的 PMK 都是独立的，即使某个用户的凭证泄露，也不会影响其他用户。

## TKIP 的局限性与 Beck-Tews 攻击

### 长期安全标准的需求 (page 28)

**TKIP 的根本问题**：尽管 TKIP 修补了 WEP 的漏洞，但由于仍然使用 **RC4 密码**，本质上不是一个理想的解决方案。

**Beck-Tews 攻击**：
- **攻击类型**：主动攻击
- **基础方法**：基于 **ChopChop** 方法恢复明文
- **前提条件**：需要支持替代的 **QoS**（Quality of Service）信道来绕过反重放保护
- **速率限制**：明文猜测的正确性通过 MIC 验证失败指示，每分钟只允许 2 次失败

### Beck-Tews 攻击：ChopChop 方法 (page 29)

**ChopChop 方法原理**：

```
原始加密数据包：
┌───────────────┬──────────┐
│ Encrypted Data│ ENC. ICV │
└───────────────┴──────────┘

删除最后一个字节：
┌───────────────┬──┐
│ Encrypted Data│? │
└───────────────┴──┘

猜测被删除字节的值并计算新的 ICV：
┌───────────────┬──────────┐
│ Encrypted Data│ New ICV  │
└───────────────┴──────────┘
         ↓
    发送并观察
         ↓
   MIC 验证失败？
   ├─ 是 → 猜测错误，重试
   └─ 否 → 猜测正确，继续下一字节
```

**工作原理**：
1. 攻击者删除加密数据包的最后一个字节
2. 猜测该字节的值并重新计算 ICV
3. 发送修改后的数据包
4. 如果 MIC 验证失败，说明猜测错误；否则猜测正确
5. 逐字节恢复明文

### Beck-Tews 攻击的实施条件 (page 30)

**正常情况下**：
- WPA 中，从过去接受的加密数据包伪造的加密包会被**丢弃**
- 原因：IV 值会被检查，防止重放攻击

**攻击窗口**：
- 支持 **IEEE 802.11e QoS** 特性的 WPA 实现可被攻击
- QoS 允许 **8 个不同的信道**用于不同的数据流
- **每个信道独立维护 TSC（Temporal Sequence Counter）计数器**

**攻击步骤**：
1. 攻击者从一个信道捕获加密数据包
2. 在**另一个 TSC 计数器较小的信道**上执行 ChopChop 攻击
3. 由于不同信道的 TSC 计数器独立，绕过了重放检测

> [!important] 攻击的核心漏洞
> 
> QoS 的多信道设计为了提高性能，每个信道独立管理序列号。但这破坏了全局的重放保护机制，攻击者可以在不同信道间重放修改过的数据包。

## 数据完整性保护机制

### Michael 消息完整性代码（MIC）算法 (page 32)

**MIC**（Message Integrity Code）是使用 "Michael" 算法计算的 **64 位消息完整性码**。

**设计约束**：
- 为已部署的传统硬件设计
- Michael 算法是 WPA 设计者能想到的**最强算法**，同时仍能在大多数旧网卡上工作
- 使用大量**交换、移位、甚至丢弃操作**

**Michael 的保护范围**：
- 保护**数据和头部**
- MIC 是**逐包**计算的哈希值
- MIC 基于**每发送者、每接收者**的基础
- MIC 基于：种子值（IVs）、目标 MAC、源 MAC 和负载

### Michael MIC 算法的工作流程 (page 33)

```
┌──────────────────┐
│ Destination MAC  │
├──────────────────┤
│   Source MAC     │
├──────────────────┤
│ 802.11 Data      │
│    Payload       │
└──────────────────┘
         ↓
    [Michael]
         ↓
   ┌──────────┐
   │ 64-bit   │
   │   MIC    │
   └──────────┘
         ↓
┌──────────────────────────────────┐
│ Data Payload │ 64-bit MIC │ ... │
└──────────────────────────────────┘
         ↓
┌────────────────────────────────────────────────┐
│ 16-bit IV │ Data Payload │ 64-bit MIC │ 32-bit ICV │
└────────────────────────────────────────────────┘
         ↓
   Per-Frame Encryption
         ↓
   Frame Transmitted
```

**流程说明**：
1. 将目标 MAC、源 MAC 和负载输入 Michael 算法
2. 计算出 64 位 MIC
3. 将 MIC 附加到数据负载后
4. 添加 16 位 IV
5. 计算 32 位 ICV（完整性校验值）
6. 对整个帧进行逐帧加密
7. 发送加密帧

### 防御重放攻击：IV 序列强制 (page 34)

由于 Michael 的不可避免的弱点，WPA 实现了 **16 位单调递增计数器**来阻止重放攻击：

**机制**：
1. 取旧 IV 的**第 1 和第 3 字节**构成序列号
2. 在重新密钥（rekey）时将**数据包序列号重置为 0**
3. 每个数据包将序列号**递增 1**
4. **丢弃任何乱序接收的数据包**

**示意图**：

```
接入点 (AP)                        无线站点 (Wireless Station)
     │                                      │
     ├────── Hdr │ Packet n ───────────────→│  接受
     │                                      │
     ├────── Hdr │ Packet n+1 ─────────────→│  接受
     │                                      │
     ├────── Hdr │ Packet n ───────────────→│  丢弃（序列号 n < n+1）
     │                                      │
```

**工作原理**：
- 接收方维护已接收的最大序列号
- 如果收到序列号小于等于已记录最大值的数据包，丢弃
- 有效防止攻击者重放旧数据包

> [!note] 为什么需要额外的序列号机制？
> 
> Michael MIC 本身只能检测数据篡改，不能防止重放。攻击者可以重放之前捕获的合法加密数据包（包含正确的 MIC），序列号强制机制确保每个数据包只能被接收一次。

## 密钥重生成机制

### TKIP 重新密钥机制 (page 35)

**定义**：指向站点和 AP 传递新的加密和完整性密钥（MIC Keys）的过程。

**WPA 密钥层次结构**：
- **主密钥**：称为 **Pairwise Master Key (PMK)**
  - 从 802.1X 密钥交换派生，或
  - 从密码短语派生（PSK 模式）
  
- **会话密钥**：统称为 **Pairwise Transient Key (PTK)**
  - 从主密钥派生
  - 用于实际的数据加密和完整性保护

**密钥刷新的意义**：
- 限制单个密钥的使用时间和数据量
- 即使攻击者破解了一个会话密钥，也不影响其他会话
- 周期性更新密钥增加攻击难度

## 总结

### WPA 的历史地位

WPA 是无线网络安全演进中的**关键过渡方案**：
- ✓ 快速响应了 WEP 的安全危机
- ✓ 实现了向后兼容，保护了用户投资
- ✓ 引入了密钥管理、完整性保护、认证框架等重要机制
- ✗ 仍然基于 RC4 密码，存在固有弱点
- ✗ Michael MIC 算法较弱，需要额外对抗措施

### WPA 的两条路径

| 方案 | 适用场景 | 安全性 | 复杂度 |
|------|----------|--------|--------|
| **WPA-PSK** | 家庭、小型办公 | 依赖密码强度，易受字典攻击 | 低 |
| **WPA + 802.1X** | 企业、政府、教育 | 强，目前无有效攻击 | 高，需要 RADIUS 服务器 |

### 关键技术点回顾

1. **TKIP**：两阶段密钥混合，为每个数据包生成独立密钥
2. **Michael MIC**：64 位消息完整性码，保护数据和头部
3. **IV 序列强制**：16 位单调递增计数器，防御重放攻击
4. **密钥层次**：PMK（主密钥）→ PTK（会话密钥）
5. **四次握手**：认证 + 密钥协商，安全性依赖密码强度
6. **802.1X + EAP**：企业模式的认证框架，每用户独立密钥
7. **Beck-Tews 攻击**：利用 QoS 多信道的 TSC 计数器独立性，绕过重放保护

### 下一步演进

WPA 的局限性（特别是 RC4 和 Michael 的弱点）促使业界开发更强的长期安全标准，最终导致 **WPA2**（基于 AES-CCMP）和 **WPA3** 的诞生。
