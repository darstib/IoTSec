## 本讲概述

本讲深入分析 WiFi 早期安全协议 WEP（Wired Equivalent Privacy，有线等效加密协议）的设计、实现与致命缺陷。WEP 是 IEEE 802.11 标准在 1999 年引入的链路层加密协议，旨在为无线网络提供与有线网络等效的安全性。然而，WEP 在设计和实现上存在多处严重漏洞，完全未能实现其机密性、完整性、访问控制等安全目标，最终于 2005 年被 IEEE 802.11 委员会正式废弃。

**核心议题**：
- 无线通信安全需求与威胁模型
- WEP 协议的加解密流程
- WEP 的根本性设计缺陷（IV 重用、弱密钥、弱完整性校验、缺乏密钥管理）
- 针对 WEP 的经典攻击（FMS 攻击、ChopChop 攻击、分片攻击）
- 无线网络安全挑战

## 1. 无线通信安全需求与威胁模型 (page 3-4)

### 1.1 无线网络面临的攻击

**物理层攻击**：
- **频率干扰（Frequency Jamming）**：通过发送干扰信号占用目标频段，阻止合法通信。技术含量不高但有效。

**MAC 层攻击**：
- **伪造解除关联消息（Spoofed Disassociation Messages）**：攻击者伪造来自 AP 的解除关联帧，强制客户端断开连接。可针对特定用户发起。

**高层协议攻击（TCP/IP 层）**：
- **SYN 洪泛攻击（SYN Flooding）**：利用 TCP 三次握手机制，发送大量伪造的 SYN 请求耗尽服务器资源，造成拒绝服务。

> [!example]+ 中间人攻击示例（Man-in-the-Middle Attack）
>
> 1. 攻击者伪造解除关联消息，使受害者与真实 AP 断开连接
> 2. 攻击者在另一个信道上建立伪造 AP，使用真实 AP 的 MAC 地址进行广播
> 3. 受害者连接到伪造 AP
> 4. 攻击者使用受害者的 MAC 地址连接到真实 AP
> 5. 攻击者成为中间人，可以监听、篡改双向流量

### 1.2 无线通信安全需求

针对上述威胁，无线通信系统需要满足以下安全目标：

| 安全需求 | 含义 |
|:---:|:---|
| **机密性（Confidentiality）** | 防止未授权方窃听通信内容 |
| **完整性（Integrity）** | 检测消息是否被篡改 |
| **真实性（Authenticity）** | 验证通信方身份 |
| **重放检测（Replay Detection）** | 防止攻击者重放先前捕获的合法消息 |
| **访问控制（Access Control）** | 限制只有授权设备可以接入网络 |
| **抗干扰（Protection against Jamming）** | 抵御物理层干扰攻击 |

## 2. WiFi 安全协议演进 (page 5)

WiFi 安全协议经历了三代演进：

1. **WEP（Wired Equivalent Privacy）**：1999 年推出，存在严重缺陷，已被废弃
2. **WPA（Wi-Fi Protected Access）**：2003 年作为 WEP 的临时替代方案
3. **WPA2（Wi-Fi Protected Access II）**：2004 年推出，基于 AES 加密，目前广泛使用

本讲重点分析 WEP 的设计与漏洞。

## 3. WEP 协议概述 (page 6-7)

### 3.1 什么是 WEP？

> [!definition]+ WEP（Wired Equivalent Privacy，有线等效加密协议）
>
> WEP 是 IEEE 802.11 标准定义的**链路层加密协议**，于 1999 年引入，旨在为无线局域网提供与有线网络相当的安全性。

**WEP 的设计目标**：
- **机密性（Confidentiality）**：加密无线传输，防止窃听
- **访问控制（Access Control）**：只允许授权设备接入网络
- **数据完整性（Data Integrity）**：检测消息是否被篡改

**然而，WEP 完全未能实现上述任何一个目标**。

### 3.2 WEP 的多种配置（WEP Flavors）

WEP 支持不同的密钥长度配置：

| 配置 | 密钥长度 | IV 长度 | 总种子长度 | 说明 |
|:---:|:---:|:---:|:---:|:---|
| **64 位 WEP** | 40 位 | 24 位 | 64 位 | 最初版本（1999） |
| **128 位 WEP** | 104 位 | 24 位 | 128 位 | 104 位 = 13 个 ASCII 字符 = 26 个十六进制字符 |
| **256 位 WEP** | 232 位 | 24 位 | 256 位 | 可用但不常见 |

所有配置都使用 **RC4 流密码**进行加密。RC4 被选择是因为它在硬件成本、速度和安全性之间取得了折中。

> [!note]+ 为什么选择 RC4？
>
> RC4 是一种流密码，计算简单、速度快，适合在资源受限的无线设备上实现。然而，RC4 存在已知的弱密钥问题，这成为 WEP 安全性的致命缺陷之一。

### 4.3 解密步骤 (page 13-14)

解密是加密的逆过程：

**步骤 1：重建密钥流**
- 从接收到的数据帧中提取 IV（明文部分）
- 将 IV 与共享密钥连接
- 使用 RC4 生成相同的密钥流

```
密钥流 = RC4(IV || Key)
```

**步骤 2：解密并验证**
- 将密钥流与密文进行 XOR 运算，恢复明文
- 使用提取的 CRC 验证消息完整性

```
明文 = 密文 ⊕ 密钥流
消息 || CRC' = 明文
验证：CRC' == CRC-32(消息)
```

## 5. WEP 的根本性缺陷 (page 15-27)

WEP 在设计和实现上存在多处致命缺陷，使其完全无法满足安全需求。以下逐一分析。

### 5.1 密钥流重用问题（Keystream Reuse）(page 16-19)

#### 问题根源

**流密码的基本原则**：*永远不要使用相同的密钥加密两次*。

然而，WEP 违反了这一原则：
- WEP 种子 = 24 位 IV + **固定的**共享密钥
- 只要使用相同的 IV 和相同的主密钥，生成的密钥流就完全相同

#### 为什么会发生密钥流重用？

1. **IV 空间太小**：24 位 IV 只有 $2^{24} \approx 1670$ 万种可能
   - 假设每个数据包 1500 字节，在 11 Mbps 速率下
   - 约 **5 小时**即可耗尽所有 IV
   
2. **生日悖论（Birthday Paradox）**：
   - 如果 IV 随机分配，发送约 **5000 个数据包**后，就有 50% 的概率出现 IV 碰撞
   
3. **顺序分配的危险**：
   - 许多实现从 0 开始顺序分配 IV
   - 设备重启后重新从 0 开始，导致 IV 必然重复

4. **主密钥很少更换**：
   - 由于缺乏密钥管理机制（后文详述），主密钥往往长期不变

#### 密钥流重用的危害

假设攻击者捕获两个使用相同密钥流加密的数据包：

```
C₁ = P₁ ⊕ K    （K 是密钥流）
C₂ = P₂ ⊕ K
```

由于 XOR 运算的可交换性：

```
C₁ ⊕ C₂ = (P₁ ⊕ K) ⊕ (P₂ ⊕ K)
        = P₁ ⊕ P₂ ⊕ K ⊕ K
        = P₁ ⊕ P₂
```

**结果**：攻击者无需知道密钥，就能得到两个明文的 XOR 结果。结合已知明文攻击（如已知的 IP 头部、ICMP、ARP 等协议结构），可以逐步恢复完整明文。

> [!example]+ 已知明文攻击示例
>
> 假设攻击者知道 P₁ 的部分内容（例如 IP 数据包的前 20 字节头部结构是已知的）：
> - 已知：P₁ 的前 20 字节
> - 捕获：C₁ 和 C₂（使用相同 IV）
> - 计算：K 的前 20 字节 = P₁[0:20] ⊕ C₁[0:20]
> - 恢复：P₂ 的前 20 字节 = K[0:20] ⊕ C₂[0:20]
>
> 通过反复试探和扩展，可以逐步恢复整个密钥流。

#### IV 的其他问题

- **IV 以明文传输**：攻击者可以轻易获取 IV，便于跟踪和分析
- **只有 24 位**：空间太小，碰撞不可避免
- **没有重用限制**：协议未规定 IV 的使用策略

### 5.2 弱密钥与弱 IV（Weak Keys and Weak IVs）(page 20)

RC4 算法存在已知的**弱密钥**问题，某些密钥更容易暴露明文与密文之间的关系。

**弱 IV 的特征**：
- 第一字节：3 到 7 之间
- 第二字节：255
- 第三字节：任意值

**统计**：
- 在 40 位 WEP 密钥配置下，约有 **9000 个弱 IV**
- 占总 IV 数量的 **约 5%**

### 5.3 弱完整性校验（Weak Integrity Check, ICV）(page 21)

WEP 使用 **CRC-32**（循环冗余校验）作为完整性校验值（ICV, Integrity Check Value）。

#### CRC-32 的缺陷

CRC-32 不是**密码学安全**的完整性保护机制：
- **设计目的**：检测随机传输错误（噪声导致的比特翻转），而非恶意篡改
- **线性特性**：CRC 是线性函数，满足 $\text{CRC}(A \oplus B) = \text{CRC}(A) \oplus \text{CRC}(B)$
- **易于计算**：任何人都可以计算和修改 CRC，无需密钥

#### 攻击后果

攻击者可以：
1. **篡改密文**：翻转密文中的特定比特
2. **修正 CRC**：利用 CRC 的线性性重新计算校验和
3. **伪装成合法消息**：接收方（AP）无法检测到篡改

这使得以下攻击成为可能：
- **重放攻击（Replay Attacks）**：重新发送先前捕获的合法数据包
- **注入攻击（Injection Attacks）**：构造并注入伪造的数据包
- **比特翻转攻击（Bit-flipping Attacks）**：精确修改特定字段（如目标 IP 地址）

> [!note]+ 密码学 MAC 与 CRC 的区别
>
> **密码学消息认证码（MAC）**（如 HMAC）：
> - 使用密钥生成，无密钥无法伪造
> - 即使修改一位，MAC 值完全改变
> - 可以防御恶意篡改
>
> **CRC-32**：
> - 无密钥，纯数学计算
> - 攻击者可以自行计算和修改
> - 只能检测随机错误，不能防御恶意攻击

### 5.4 缺乏密钥管理（Key Management）(page 22, 24)

WEP 协议**完全没有定义任何密钥管理或交换框架**。

#### 实际部署中的问题

1. **手动分发**：主密钥必须手动配置到每个设备
2. **静态配置**：密钥长期不变
3. **共享密钥**：所有设备使用相同的密钥（通常是 4 个全局共享密钥之一）
4. **易于记忆**：为了便于输入，密钥往往选择简单字符串

#### 密钥输入方式

- **直接设置十六进制数据**：输入 26 位十六进制字符（104 位密钥）
- **ASCII 字符串生成器**（非标准，但广泛使用）：
  - 输入易记的 ASCII 字符串
  - 通过某种算法转换为密钥材料
  - 不同厂商的算法可能不兼容

#### 后果

- 密钥很少更换，给攻击者充足的时间进行密码分析
- 弱密钥（如简单字符串生成的密钥）容易被字典攻击破解
- 一旦密钥泄露，整个网络失去保护

### 5.5 缺乏重放保护（No Replay Protection）(page 26)

WEP 没有对 IV 进行重放检测：
- 攻击者可以捕获合法数据包并重放
- 常见攻击：**ARP 重放攻击**（用于快速收集大量相同 IV 的数据包，加速密钥破解）

### 5.6 用户认证缺陷（User Authentication）(page 25)

#### 共享密钥认证（Shared Key Authentication）协议

WEP 使用一个四步握手协议来认证站点：

1. 发起方请求共享密钥认证
2. 响应方发送随机挑战文本（明文）
3. 发起方用 WEP 加密挑战文本并返回
4. 响应方验证加密结果，完成认证

#### 协议的致命缺陷

这个协议本应证明发起方知道正确的密钥，但存在严重漏洞：

- 攻击者捕获认证过程
- 获得：**已知明文**（挑战文本）和对应的**密文**
- 计算：`密钥流 = 明文 ⊕ 密文`
- 结果：**无需知道密钥，就能提取密钥流**

一旦获得密钥流，攻击者可以：
- 使用该密钥流加密自己的挑战响应
- 成功通过认证
- 接入网络（尽管仍不知道主密钥，但可以进行后续攻击）

### 5.7 缺乏消息认证（Lack of Message Authentication）(page 27)

除了前述缺陷，WEP 还存在以下问题：

**信标消息（Beacon Messages）未认证**：
- AP 定期广播的信标帧没有认证机制
- 攻击者可以伪造信标消息，诱导客户端连接到伪造 AP（邪恶双胞胎攻击，Evil Twin Attack）

**管理帧未认证**：
- **解除关联（Disassociate）** 和 **解除认证（Deauthenticate）** 消息未认证
- 攻击者可以伪造这些帧，强制合法用户断开连接
- 导致**拒绝服务（DoS）**攻击

## 6. WEP 的经典攻击 (page 28-36)

针对 WEP 的多重缺陷，研究人员和攻击者开发了多种攻击方法。以下介绍四种代表性攻击。

### 6.1 暴力破解与字典攻击（Brute Force / Dictionary Attack）(page 29-30)

#### 原理

WEP 的"安全性"依赖于通过暴力破解发现密钥的困难性。然而：
- **40 位密钥**：只有 $2^{40} \approx 1.1$ 万亿种可能，现代计算能力下可在合理时间内遍历
- **字典攻击**：使用常见密钥字典（而非尝试所有可能），进一步缩短时间

#### 攻击步骤

1. **捕获**至少 2 个使用 WEP 加密的数据包（相同或不同 IV）
2. **尝试解密**：使用捕获的 IV 和候选密钥，通过 RC4 生成密钥流，解密密文
3. **验证 ICV**：检查解密后的 CRC-32 是否正确
4. **可选验证**：用第二个数据包再次验证密钥

#### 时间复杂度

- 40 位密钥：**少于一个月**可遍历所有可能
- 使用字典攻击：时间进一步缩短
- 随着计算能力提升和 GPU 加速，现代设备可在**数小时至数天**内破解

### 6.2 FMS 攻击（Fluhrer-Martin-Shamir Attack）(page 31)

这是最著名和最有效的 WEP 攻击之一，由 Fluhrer、Martin 和 Shamir 在 2001 年提出。

#### 理论基础

- RC4 存在一类**弱密钥**，使用这些密钥时，密钥流的前几个字节与密钥本身存在统计相关性
- 如果攻击者知道密钥流的前 2 个字节，就能推断出 RC4 密钥

#### WEP 中的已知信息

- WEP 数据包的前 8 字节是**已知的 SNAP-SAP 头部**：`AA AA 03 00 00 00 08 ??`
  - 前 6 字节固定
  - 第 7 字节标识以太网类型（通常是 `08`，表示 IP）
  - 第 8 字节部分可预测
- 通过 `密钥流 = 明文 ⊕ 密文`，攻击者可以获得前几个字节的密钥流

#### 攻击步骤

1. **收集弱 IV 数据包**：捕获大量使用弱 IV 的数据包（约 5% 的 IV 是弱 IV）
2. **提取密钥流前缀**：利用已知的 SNAP-SAP 头部，计算每个数据包的密钥流前几字节
3. **统计分析**：利用 RC4 的弱密钥特性，通过统计方法逐字节恢复密钥
4. **恢复完整密钥**：重复上述过程，逐步恢复所有密钥字节

#### 实际效果

- **AirSnort** 工具实现了该攻击
- 收集约 **20,000 个数据包**后即可恢复密钥
- 在 11 Mbps 速率下，仅需 **约 11 秒**

> [!note]+ FMS 攻击的意义
>
> FMS 攻击证明了 WEP 不仅存在理论缺陷，更可以被实际、快速地破解。这直接导致了 WEP 被废弃的命运。

### 6.3 ChopChop 攻击（KoreK ChopChop Attack）(page 32-33)

ChopChop 攻击不恢复密钥，而是直接**解密消息内容**，利用了 ICV（CRC-32）的弱完整性校验。

#### 核心思想

利用 AP 的解密反馈作为"预言机（Oracle）"：
- 发送修改后的数据包给 AP
- 如果 AP 回复，说明 CRC 校验通过，猜测正确
- 如果 AP 不回复，说明 CRC 校验失败，猜测错误

#### 攻击步骤

1. **捕获一个加密数据包**
2. **截断最后一个字节**
3. **猜测被删除字节的明文值**
4. **重新计算 CRC**：根据猜测值调整 CRC（利用 CRC 的线性性）
5. **发送修改后的数据包给 AP**
6. **观察 AP 响应**：
   - 如果 AP 回复 → 猜测正确
   - 如果 AP 无响应 → 猜测错误，尝试下一个值
7. **重复上述过程**，逐字节向前解密整个消息

#### 示意图（简化）

```
原始密文：[加密数据........................] [加密 ICV]
第一轮： [加密数据.....................?]  猜测最后一个字节
         计算新 ICV 并发送 → 观察 AP 响应
成功后：
第二轮： [加密数据..................??]    猜测倒数第二个字节
         ...
```

### 6.4 分片攻击（Fragmentation Attack）(page 35-36)

分片攻击利用 802.11 和 IP 协议的分片机制，从少量已知明文中提取足够的密钥流，进而发送任意数据。

#### 背景：什么是分片？

- **分片（Fragmentation）**：将大数据包切分为多个小数据包传输
- **目的**：在噪声环境中提高传输成功率（单个小包丢失的代价小于整个大包丢失）
- **802.11 规定**：每个数据包最多可分为 **16 个片段**
- **WEP 加密行为**：使用**相同的 IV**独立加密每个片段

#### 攻击原理

**已知明文来源**：几乎所有加密数据包都包含 **LLC/SNAP 头部**（8 字节）：

```
AA AA 03 00 00 00 08 ??
│  │  │  └─────┬────┘  │
│  │  │      组织代码  │
│  │  │              以太网类型（08 表示 IP，06 表示 ARP）
│  │  控制字段
│  SSAP
DSAP
```

- 前 6 字节固定：`AA AA 03 00 00 00`
- 第 7 字节：`08`（IP）或 `06`（ARP）
- 第 8 字节部分可预测

**关键操作**：
1. 捕获任意一个加密数据包
2. 利用已知的 8 字节 LLC/SNAP 头部，计算密钥流前 8 字节：
   ```
   密钥流[0:8] = 明文[0:8] ⊕ 密文[0:8]
   ```
3. 构造任意数据（减去 4 字节 CRC 标签后）：$(8 - 4) \times 16 = 64$ 字节
4. 将自定义数据分为 16 个片段，每个片段 4 字节
5. 使用提取的 8 字节密钥流加密这些片段
6. 发送伪造数据包

#### 攻击效果

- 从一个合法数据包中提取 8 字节密钥流
- 可以发送最多 **64 字节的任意数据**
- 无需知道完整密钥
- 可用于注入 ARP 请求、TCP 连接等攻击流量

> [!example]+ 分片攻击的实际应用
>
> 攻击者可以：
> - 注入伪造的 ARP 请求，毒化 ARP 缓存
> - 发送 TCP SYN 包，进行端口扫描
> - 触发大量响应流量，收集更多数据包用于其他攻击（如 FMS 攻击）

## 7. WEP 的防御措施（不可靠）(page 37)

虽然 WEP 存在根本性缺陷，一些"防御"措施曾被提出，但都无法真正解决问题：

| 措施 | 局限性 |
|:---|:---|
| **增加密钥长度**（如 128 位、256 位） | 对 FMS 攻击无效；只能延缓暴力破解，不能阻止 |
| **移除弱 IV** | 实现复杂；IV 空间已很小，移除 5% 进一步加剧碰撞 |
| **频繁更换密钥** | 使用 **EAP（可扩展认证协议）**动态分发密钥 | 增加复杂性，但无法解决 IV 重用、弱 ICV 等根本问题 |
| **部署入侵检测系统（IDS）** | 只能检测异常流量，无法阻止密码分析攻击 |
| **厂商专有"增强 WEP"** | 不兼容标准；通常只是表面修补，仍有漏洞 |

> [!attention]+ 根本教训
>
> **WEP 必须被彻底抛弃，而不是修修补补**。任何基于 WEP 框架的"增强"都无法解决其设计上的根本缺陷。

## 8. WEP 的历史与现状 (page 38)

- **1999 年**：WEP 随 IEEE 802.11 标准发布
- **2001 年**：Fluhrer、Martin、Shamir 发表 FMS 攻击论文，WEP 安全性被彻底打破
- **2003 年**：WiFi 联盟推出 **WPA（Wi-Fi Protected Access）**作为临时替代方案
- **2004 年**：**WPA2** 正式发布，基于 AES 加密的 CCMP 协议
- **2005 年**：IEEE 802.11 委员会**正式宣布 WEP 废弃（Deprecated）**
- **2008 年**：美国某大学调查显示，仍有 **30% 的用户使用 WEP**
- **今天**：许多老旧网络仍在使用 WEP，成为严重的安全隐患

**攻击的演化**：
- 早期攻击需要数天收集数据
- 现代工具（如 Aircrack-ng 套件）可在**数分钟**内破解 WEP

> [!summary]+ WEP 的历史教训
>
> WEP 是密码学工程中的经典反面教材，展示了以下错误如何导致系统性失败：
> 1. **违反密码学基本原则**（流密码密钥重用）
> 2. **选择非密码学安全的原语**（CRC-32 而非 HMAC）
> 3. **缺乏前瞻性设计**（IV 空间过小）
> 4. **忽视密钥管理**（手动静态配置）
> 5. **认证协议设计失误**（泄露已知明文-密文对）

## 9. 无线网络安全的根本挑战 (page 39-43)

WEP 的失败暴露了无线网络相比有线网络面临的固有安全挑战。

### 9.1 缺乏物理保护

**有线网络**：
- 攻击者需要物理接入网络设备（插入网线、接触路由器）
- 设备通常位于受保护的建筑物内

**无线网络**：
- 信号穿墙传播，覆盖范围远超建筑物边界
- 攻击者可在停车场、街道、邻近建筑发起攻击
- 物理连接被逻辑关联取代，难以控制"谁在范围内"

### 9.2 广播特性

无线通信本质是**广播**：
- **接收**：范围内任何设备都能接收到传输
- **发送**：任何设备都能发送信号，干扰合法通信

### 9.3 具体威胁

| 威胁 | 描述 |
|:---|:---|
| **窃听（Eavesdropping）** | 被动监听，极易实施，几乎不可能检测；默认情况下所有内容（用户名、密码、数据）均明文传输 |
| **中间人攻击（MITM）** | 伪造 AP，拦截并篡改双向流量 |
| **拒绝服务（DoS）** | 物理层干扰、伪造解除关联消息、SYN 洪泛 |

### 9.4 窃听的易实施性与危害 (page 43)

**窃听特点**：
- **易于实施**：互联网上有大量免费工具（网络嗅探器、协议分析器、密码收集器）
- **几乎不可检测**：被动监听不会在网络中留下痕迹
- **远距离攻击**：使用定向天线，可从**数公里外**窃听流量

**默认风险**：
- 大多数应用协议**默认不加密**（HTTP、FTP、Telnet、SMTP）
- 即使 WEP 被破解，用户名、密码、会话令牌等敏感信息直接暴露

> [!attention]+ 无线网络的安全默认应该是"零信任"
>
> 应当假设**所有无线传输都可能被窃听**，因此：
> 1. 使用端到端加密（HTTPS、VPN、SSH）
> 2. 不在无线网络上传输明文敏感信息
> 3. 采用强认证机制（双因素认证）

## 10. 总结与关键要点

### 10.1 WEP 的设计缺陷回顾

| 缺陷类型 | 具体问题 | 后果 |
|:---:|:---|:---|
| **密钥流重用** | IV 空间过小（24 位）；主密钥静态 | 5 小时耗尽 IV；生日攻击 5000 包后碰撞；已知明文攻击可恢复密钥流 |
| **弱密钥** | RC4 存在 9000 个弱 IV（约 5%） | FMS 攻击可快速恢复密钥（20,000 包，11 秒） |
| **弱完整性** | CRC-32 线性、非密码学安全 | 比特翻转攻击；注入攻击；重放攻击 |
| **无密钥管理** | 手动静态配置；全局共享 | 密钥很少更换；易被字典攻击 |
| **认证缺陷** | 共享密钥认证泄露密钥流 | 攻击者可在不知密钥的情况下通过认证 |
| **无重放保护** | 无 IV 序列号检查 | ARP 重放攻击加速密钥破解 |
| **管理帧未认证** | 信标、解除关联消息无保护 | 邪恶双胞胎攻击；DoS 攻击 |

### 10.2 典型攻击总结

| 攻击类型 | 原理 | 所需资源 | 效果 |
|:---:|:---|:---:|:---|
| **暴力破解** | 遍历所有可能密钥 | 2+ 数据包，计算能力 | 40 位密钥：<1 月；字典攻击更快 |
| **FMS 攻击** | 利用 RC4 弱密钥统计特性 | ~20,000 弱 IV 数据包 | 恢复完整密钥（11 秒） |
| **ChopChop 攻击** | 利用 CRC 弱校验，逐字节猜测明文 | 1 个数据包，AP 响应 | 解密消息内容（不恢复密钥） |
| **分片攻击** | 从已知 SNAP 头提取密钥流 | 1 个数据包 | 注入 64 字节任意数据 |

### 10.3 关键教训

1. **流密码的金科玉律**：*永远不要用相同的密钥加密两次*。WEP 的 IV 设计违反了这一原则。

2. **完整性需要密码学 MAC**：CRC 只能检测随机错误，不能防御恶意篡改。应使用 HMAC 或认证加密模式（如 AES-GCM）。

3. **密钥管理不是可选项**：没有密钥分发和更新机制，静态密钥成为整个系统的单点失败。

4. **认证协议需要谨慎设计**：WEP 的共享密钥认证反而泄露了已知明文-密文对，成为攻击的帮手。

5. **安全需要整体设计**：WEP 试图修补无线网络的安全问题，但各个组件（加密、完整性、认证）之间缺乏协调，导致系统性失败。

6. **不要修补根本缺陷的系统**：WEP 的问题不是某个参数设置不当，而是架构本身存在根本缺陷。增加密钥长度、移除弱 IV 等措施都是治标不治本。

### 10.4 WEP 之后的发展

| 协议 | 年份 | 加密算法 | 完整性保护 | 密钥管理 | 状态 |
|:---:|:---:|:---|:---|:---|:---|
| **WEP** | 1999 | RC4（流密码） | CRC-32 | 无 | 已废弃（2005） |
| **WPA** | 2003 | RC4 + TKIP | Michael | 802.1X / PSK | 过渡方案，已淘汰 |
| **WPA2** | 2004 | AES-CCMP | CBC-MAC | 802.1X / PSK | 当前主流 |
| **WPA3** | 2018 | AES-CCMP / GCMP | CBC-MAC / GCM | SAE（同时认证等价） | 最新标准 |

**WPA2/WPA3 的改进**：
- 使用 **AES（Advanced Encryption Standard）**分组密码，而非 RC4 流密码
- 采用 **CCMP（Counter Mode with CBC-MAC Protocol）**提供认证加密
- 支持 **802.1X** 企业级认证（动态密钥分发）
- **每个数据包使用唯一的密钥**（通过密钥层次结构和包计数器实现）
- **防重放保护**（包序列号检查）
- **管理帧保护（PMF）**（WPA3 强制，WPA2 可选）

### 10.5 无线安全的最佳实践

即使在 WPA2/WPA3 环境下，也应遵循以下原则：

1. **多层防御**：
   - 网络层：使用 WPA2/WPA3
   - 传输层：使用 TLS/SSL（HTTPS）
   - 应用层：使用端到端加密

2. **强认证**：
   - 避免弱密码（WPA2-PSK）
   - 企业环境使用 802.1X
   - 启用双因素认证

3. **网络隔离**：
   - 访客网络与内部网络隔离
   - IoT 设备单独 VLAN

4. **监控与审计**：
   - 部署无线 IDS/IPS
   - 检测伪造 AP
   - 记录连接日志

5. **及时更新**：
   - 固件更新
   - 禁用已废弃协议（WEP、WPA）

### WiFi 的定义

**WiFi** 是指任何遵循 **IEEE 802.11 标准**的无线网络系统：
- 由 **IEEE（电气和电子工程师协会）**开发，1997 年首次发布
- 术语"WiFi"由 **WiFi 联盟（Wi-Fi Alliance）**商业化推广
- 拼写变体：Wi-Fi、WiFi、Wifi、wifi（均指同一技术）

### 802.11 标准

802.11 是一套定义**无线局域网（WLAN）**计算机通信的标准，由 **IEEE LAN/MAN 标准委员会（IEEE 802）**维护。

**工作频段**：
- **2.4 GHz**：802.11b/g/n/ax（WiFi 4/5/6）
- **5 GHz**：802.11a/n/ac/ax（WiFi 5/6）
- **6 GHz**：802.11ax（WiFi 6E）

**标准演进**：
- 802.11（1997）：2 Mbps
- 802.11b（1999）：11 Mbps，2.4 GHz
- 802.11a（1999）：54 Mbps，5 GHz
- 802.11g（2003）：54 Mbps，2.4 GHz
- 802.11n（2009）：WiFi 4，最高 600 Mbps，MIMO
- 802.11ac（2013）：WiFi 5，最高 6.9 Gbps，MU-MIMO
- 802.11ax（2019）：WiFi 6/6E，最高 9.6 Gbps，OFDMA

### 4.1 加密涉及的组件

| 组件 | 含义 |
|:---|:---|
| **消息（Message）** | 待加密的明文数据 |
| **CRC（循环冗余校验）** | 用于验证消息完整性的校验和，使用 CRC-32 多项式计算 |
| **明文（Plaintext）** | 消息 + CRC 的组合 |
| **初始化向量（IV, Initialization Vector）** | 24 位随机数，用于生成不同的密钥流 |
| **密钥（Key）** | 40 位或 104 位的共享密钥 |
| **密钥流（Keystream）** | 由 IV + Key 通过 RC4 算法生成，用于加密明文 |
| **密文（Ciphertext）** | 加密后的数据 |

### 4.2 加密步骤

**步骤 1：计算完整性校验值（ICV）**
- 对消息使用 **CRC-32** 多项式计算校验和
- 将 CRC 附加到消息后，形成明文

```
明文 = 消息 || CRC-32(消息)
```

**步骤 2：生成密钥流（Keystream）**
- 将 24 位 **IV** 与密钥连接，形成 64 位或 128 位的 RC4 种子
- 使用 RC4 算法生成与明文等长的密钥流

```
种子 = IV (24 bits) || Key (40 或 104 bits)
密钥流 = RC4(种子)
```

**步骤 3：加密明文**
- 将明文与密钥流进行 **异或（XOR）**运算，得到密文
- 将 IV 以明文形式附加在密文前面

```
密文 = 明文 ⊕ 密钥流
发送数据 = IV || 密文
```

> [!attention]+ IV 是以明文形式传输的！
>
> IV 不加密直接附加在数据包前面，这是为了让接收方能够重建相同的密钥流进行解密。然而，这也意味着攻击者可以轻易获取 IV，为后续攻击提供了便利。
