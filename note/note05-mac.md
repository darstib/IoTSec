# 介质访问控制（MAC）协议

## 介质访问控制（MAC）协议

### 多址接入信道 (page 3)

在无线网络中，每个节点通过与其他节点共享的通道与接入点（AP）或基站（BS）进行通信。这种共享性带来了一个核心挑战：

- **信道共享特性**：一个节点的传输可以被其他节点接收
- **干扰问题**：当两个或多个节点同时传输时，会发生信号干扰，导致传输失败

> [!definition]+ 介质访问控制（MAC）协议
>
> MAC 协议是一种分布式算法，用于决定节点如何共享信道，即决定节点何时可以发射。
>
> **核心矛盾**：关于信道共享的协商本身也需要使用信道，这就像"用信道来决定如何使用信道"的自指问题。

### 理想的多址接入协议 (page 4)

一个理想的 MAC 协议应该满足以下特性：

1. **高效利用带宽**
   - 当仅有一个节点想传输时，能够以全速率 $R$ bps 传输
   - 当 $M$ 个节点想传输时，每个节点平均以 $R/M$ bps 的速率传输

2. **完全去中心化**
   - 不需要特殊节点来协调通信
   - 不需要对时钟、时间槽做同步
   - 提高系统鲁棒性，避免单点故障

### MAC 协议分类 (page 5)

根据信道分配策略，MAC 协议可分为三大类：

1. **信道分割（Channel Partitioning）**
   - 把信道切分为更小的部分（时间槽、频率或码片）
   - 把一小部分分配给单个节点独家使用
   - 典型协议：TDMA、FDMA、CDMA

2. **随机访问（Random Access）**
   - 信道没有被预先分割，允许碰撞发生
   - 需要从碰撞中"恢复"
   - 典型协议：ALOHA、CSMA/CD、CSMA/CA

3. **轮流（Taking Turns）**
   - 严格协调共享访问以避免碰撞
   - 典型协议：轮询（Polling）、令牌传递（Token Passing）

## 信道分割 MAC 协议

### 时分多址（TDMA）(page 7)

**基本原理**：将时间划分为固定长度的时间槽，节点轮流访问信道。

**工作机制**：
- 每个节点在每轮都有固定长度的时间槽
- 节点只能在分配给自己的时间槽内传输
- 未使用的时间槽被闲置

> [!example]+ TDMA 示例
>
> 在一个 6 节点的局域网中：
> - 节点 1、3、4 有数据包需要发送
> - 节点 2、5、6 的时间槽被闲置，造成资源浪费

**优缺点分析**：

✓ **优点**：消除碰撞，传输可预测

✗ **缺点**：
- 信道划分为 $N$ 个时间槽，每个用户一个
- 在低占空比用户和轻负载时效率低下
- 需要节点间时间同步

### 频分多址（FDMA）(page 8)

**基本原理**：将信道频谱划分为多个频段，每个节点被分配一个固定的频段。

**工作机制**：
- 每个节点独占一个频段进行通信
- 频段内未使用的时间被闲置
- 不同频段之间相互独立

> [!example]+ FDMA 示例
>
> 在一个 6 节点的局域网中：
> - 节点 1、3、4 使用各自的频段传输
> - 节点 2、5、6 的频段空闲

**优缺点分析**：

✓ **优点**：简单、无碰撞、无需同步

✗ **缺点**：
- 频谱资源被静态分配
- 空闲用户的频段浪费

### 码分多址（CDMA）(page 9)

**基本原理**：每个用户分配独立的地址码（码片序列），所有用户共享同一频段。

**工作机制**：
- 每个用户拥有属于自己的码片序列（地址码）来编码数据
- **编码过程**：$\text{编码后的信号} = (\text{原始数据}) \times (\text{码片序列})$
- **解码过程**：编码后的信号与码片序列的内积

**核心特性**：
- 允许多个用户"共存"，以最小的干扰同时传输
- **前提条件**：各用户的码片序列必须是正交的（orthogonal）

> [!note]+ 正交码片序列
>
> 如果两个码片序列 $C_1$ 和 $C_2$ 满足：
> $$C_1 \cdot C_2 = 0$$
> 则称它们正交。正交性确保了不同用户的信号互不干扰，接收方可以准确解码出目标信号。

**应用场景**：主要用于无线广播信道，如蜂窝网络、卫星通信等。

### 信道分割的控制方式 (page 10)

信道分割协议的资源分配可以通过两种方式实现：

1. **中心化协调**
   - 由中心化的协调节点分配时间/频率/码片资源
   - 典型应用：IEEE 802.11 架构模式、蜂窝网络、电缆调制解调器（Cable Modem）
   - 优点：分配高效、冲突少
   - 缺点：存在单点故障风险

2. **分布式共识协议**
   - 节点广播它们要使用的时间/频率/代码以及持续时间
   - 通过单独的控制信道进行协商
   - 典型应用：Ad-hoc 网络、移动自组织网络（MANET）
   - 优点：去中心化、鲁棒性高
   - 缺点：协商开销大

## 随机访问 MAC 协议

### 随机访问协议概述 (page 11)

**核心思想**：当节点有数据包要发送时，以全信道速率 $R$ bps 传输，节点间没有预先的协商。

**关键问题**：
- 两个或多个节点同时传输会导致"碰撞"（collision）
- 随机访问 MAC 协议需要解决：
  - 如何检测碰撞
  - 如何从碰撞中恢复（如：通过延迟的重新传输）

### 随机访问协议的类型 (page 12)

常见的随机访问 MAC 协议包括：

1. **ALOHA**：最简单的随机访问协议
2. **时隙 ALOHA（Slotted ALOHA）**：ALOHA 的改进版本
3. **载波监听多路访问（CSMA）**：传输前监听信道
4. **CSMA/CD**：CSMA 加碰撞检测，用于有线网络（如以太网）
5. **CSMA/CA**：CSMA 加碰撞避免，用于无线网络（如 WiFi）
6. **带有 RTS/CTS 的 CSMA/CA**：解决隐藏终端和显露终端问题

### ALOHA 协议 (page 13)

**工作机制**：
- 每当一个节点有数据时，它就立即发送
- 发送方通过监听目的站的广播，判断传输是否成功或遭遇碰撞
- 如果发生碰撞，发送方在某个随机时间后重新发送

**性能分析**：
- **最高吞吐量**：约 **18%**
- 原因：碰撞窗口大（任何时间的传输都可能与其他传输重叠）

> [!example]+ ALOHA 碰撞示例
>
> - 节点 1 发送数据包
> - 节点 2 和节点 3 在不同时刻也开始发送
> - 三个节点的传输时间有重叠，导致碰撞
> - 各节点检测到碰撞后，等待随机时间后重传

### 时隙 ALOHA（Slotted ALOHA）(page 14)

**改进思路**：通过时间同步减少碰撞窗口。

**工作机制**：
- 时间被划分为同等大小的时间段（时隙）
- 节点仅在时间槽之初开始传输帧
- 节点之间是同步的
- 如果 2 个或更多的节点在同一时隙中传输，所有节点都会检测到碰撞

**优缺点分析** (page 15)：

✓ **优点**：
- 单个活动节点能以信道的全部速率连续传输
- 高度去中心化：只有节点的时间槽需要保持同步
- 吞吐量比纯 ALOHA 高（约 37%）

✗ **缺点**：
- 碰撞，浪费时间槽
- 空闲的时间槽
- 需要时间同步

### 载波监听多路访问（CSMA）(page 16)

**改进思路**：在传输数据包前先监听信道（"先听后说"）。

**工作机制**：
- **载波监听**：仅当检测到信道空闲时才开始传输
- 如果信道繁忙，则推迟传输

**为什么仍会发生碰撞？**

由于非零的传播延迟，两个节点可能同时检测到信道空闲并开始传输，导致碰撞。

> [!note]+ 传播延迟的影响
>
> 碰撞的概率随着传播延迟而增加。即使节点 A 已经开始传输，节点 B 在距离较远处可能还未感知到载波，从而误判信道空闲并开始传输。

**问题**：即使发生了碰撞，节点仍继续传输完整个数据包，导致信道容量的完全浪费。

### CSMA/CD（碰撞检测）(page 17)

**改进思路**：如果检测到碰撞，立即停止正在进行的传输。

**工作机制**：

1. **碰撞检测（Collision Detection）**：检测碰撞的发生
2. **停止传输**：一旦检测到碰撞，立即停止传输
3. **随机退避**：如果发生碰撞，所有节点都会退后，等待一个随机的时间后重试

**帧时间要求**：
- 每个帧的时间应至少是检测碰撞时间的两倍
- 即：$T_{\text{frame}} \geq 2 \times T_{\text{propagation}}$

**无线网络中的问题**：

碰撞检测（"边听边说"）在无线网络中不起作用，原因：
- 无线信号衰减严重，接收信号功率远小于发送功率
- 节点在发送时无法有效监听到微弱的碰撞信号
- 碰撞的成本很高（只有在发送了整个数据包而没有收到 ACK 后才会被发现）

### CSMA/CA（碰撞避免）(page 18)

**设计目标**：专为无线网络设计，通过预防机制降低碰撞概率。

**核心思路**：
- 当检测到介质空闲时，在传输之前，通过启动随机回退（backoff）定时器，将碰撞的机会降到最低
- 使用空闲信道评估（Clear Channel Assessment, CCA）进行载波监听

**工作机制**：

1. **载波监听**：
   - 载波存在 ⟹ 不传输（延迟或回退）
   - 没有载波 ⟹ 或许可以传输（等待 DIFS）

2. **帧间间隙（Inter-Frame Spaces, IFS）**：
   - **DIFS**（Distributed Inter-Frame Spacing）：分布式帧间间隙
   - 用于实现不同的固定优先级别（用于 QoS）

3. **随机回退机制**：
   - 当信道繁忙时，启动随机回退定时器
   - 只有当信道空闲时，定时器才会倒计时
   - 定时器到点后发送数据

### CSMA/CA 基本流程 (page 19)

**发送方流程**：

1. 如果持续 DIFS 时间检测到信道空闲
   - 发送整个数据帧（没有碰撞检测）

2. 如果检测到信道繁忙
   - 开始随机回退一段时间
   - 当信道空闲时定时器才会倒计时
   - 定时器到点后发送数据

3. 如果没有收到 ACK
   - 增加随机回退的时间
   - 重复步骤 2

**接收方流程**：

- 如果成功收到数据帧
  - 在 SIFS（Short Inter-Frame Spacing）后返回 ACK

> [!note]+ SIFS vs DIFS
>
> - **SIFS**：短帧间间隙，优先级高，用于 ACK、CTS 等控制帧
> - **DIFS**：分布式帧间间隙，优先级较低，用于数据帧
> - SIFS < DIFS，确保 ACK 能优先发送

### CSMA/CA 工作流程示例 (page 20-26)

> [!example]+ CSMA/CA 时序示例
>
> 假设有节点 A、B、C 想要发送数据：
>
> **时刻 1**：节点 A 检测信道空闲，等待 DIFS 时间
>
> **时刻 2**：DIFS 时间结束，节点 A 开始发送数据帧
>
> **时刻 3**：节点 B 和 C 检测到信道繁忙，启动各自的随机回退定时器
>
> **时刻 4**：节点 A 发送完成，接收方在 SIFS 后发送 ACK
>
> **时刻 5**：节点 B 和 C 继续倒计时回退定时器（仅当信道空闲时）
>
> **时刻 6**：节点 B 的定时器先到期，开始发送数据
>
> **时刻 7**：节点 C 检测到信道繁忙，暂停定时器，等待信道再次空闲后继续倒计时

### 隐藏终端问题（Hidden Node）(page 27)

**问题描述**：

节点 A 不知道节点 B 目前正忙于接收来自节点 C 的信息，因此可能开始自己的传输，造成碰撞。

**场景分析**：
- 节点 C 正在向节点 B 发送数据
- 节点 A 在节点 C 的传输范围之外，无法检测到载波
- 节点 A 误判信道空闲，开始向节点 B 传输
- 在节点 B 处发生碰撞

**根本原因**：载波监听的局限性——节点只能感知自己传输范围内的载波。

### 显露终端问题（Exposed Node）(page 28)

**问题描述**：

节点 B 想向节点 C 传输，但错误地认为这将干扰 A 向 D 的传输，所以 B 取消了传输。

**场景分析**：
- 节点 A 正在向节点 D 发送数据
- 节点 B 检测到来自节点 A 的载波
- 节点 B 想向节点 C 传输，但因检测到载波而放弃
- 实际上，B 向 C 的传输不会干扰 A 向 D 的传输（两者在不同的接收范围）

**影响**：效率降低，本可以并行传输的机会被浪费。

### 带有 RTS/CTS 的 CSMA/CA (page 29-30)

**设计目标**：通过握手机制解决隐藏终端和显露终端问题。

**工作机制**：

1. **RTS（Request to Send）请求发送**
   - 发送方发送 RTS，表明它想使用多长时间的介质
   - RTS 包含预期的传输时间

2. **CTS（Clear to Send）允许发送**
   - 接收方以 CTS 作为回复，呼应预期的传输时间
   - CTS 广播到接收方的邻居节点

3. **数据传输**
   - 发送方收到 CTS 后开始传输数据

4. **ACK 确认**
   - 接收方成功接收后发送 ACK
   - 所有节点都必须等待接收方的 ACK，然后再尝试发送

**解决隐藏终端问题**：
- 节点 A 和 C 都想发给节点 B
- 节点 A 率先发送 RTS
- 来自 B 的 CTS 被 C 收到
- 节点 C 收到 CTS 后等待，避免碰撞

**解决显露终端问题**：
- 节点 B 想发送至 A，C 想发送到另一个终端
- 节点 C 听到 B 的 RTS，但收不到 A 的 CTS
- 节点 C 不需要等待，因为它的传输不会干扰 B 向 A 的通信

## 轮流 MAC 协议

### 轮询（Polling）和令牌传递（Token Passing）(page 31)

#### 轮询（Polling）

**工作机制**：
- 主节点"邀请"从属节点依次进行传输
- 中心化控制，按固定顺序轮询

**问题**：
- 轮询的开销（主节点需要逐个询问）
- 延迟（节点需要等待被轮询到）
- 单点故障（主节点）

#### 令牌传递（Token Passing）

**工作机制**：
- 控制令牌从一个节点依次传递到下一个节点
- 持有令牌的节点可以传输数据
- 传输完毕后将令牌传递给下一个节点

**问题**：
- 令牌开销（令牌消息本身占用带宽）
- 延迟（需要等待令牌到达）
- 单点故障（令牌丢失或损坏）

### MAC 协议总结 (page 32)

**如何处理共享介质的问题？**

1. **信道分割**（通过时间、频率或码片）
   - TDMA、FDMA、CDMA
   - 优点：无碰撞，适合高负载
   - 缺点：低负载时效率低

2. **随机访问**（动态）
   - ALOHA、S-ALOHA、CSMA
   - 碰撞检测：在有线网络中容易实现（CSMA/CD 在以太网中使用）
   - 碰撞避免：在无线网络中使用（CSMA/CA）

3. **轮流**
   - 轮询、令牌传递
   - 优点：公平、有序
   - 缺点：存在单点故障风险
