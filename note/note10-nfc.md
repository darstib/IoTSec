## 1. NFC 基础概念

### 1.1 什么是 NFC

**近场通信（Near Field Communication, NFC）** 是一组短距离、高频的射频识别（RFID）数据传输标准，工作频率为 **13.56 MHz**，典型通信距离在 **4 cm 到 10 cm** 之间。

NFC 的核心参数：

| 参数 | 取值 |
|------|------|
| 工作频率 | 13.56 MHz |
| 通信距离 | 4 cm – 10 cm |
| 数据速率 | 106 / 212 / 424 kbit/s |
| 底层标准 | ISO 14443、ISO 15693、ISO 18092 |

NFC 通信发生在两个设备之间，角色明确：

- **发起方（Initiator）**：例如阅读器，主动发起通信
- **目标方（Target）**：例如标签，响应发起方的请求

> [!note]+ NFC Forum 的角色
>
> **NFC Forum** 是制定 NFC 互操作性和应用规范的行业组织，它定义了：
> - 设备之间的**互操作性（Interoperability）** 标准
> - NFC 应用规范（NFC application specification）
>
> NFC Forum 并不直接定义加密或安全机制，这一点对后续安全分析至关重要。

### 1.2 NFC 的三种工作模式 (page 34)

NFC 设备有三种主动工作模式，分别对应不同的应用场景和底层标准：

**1. 读写模式（Reader / Writer Mode，PCD，ISO 14443）**
- 主动设备（如手机）发射功率，作为阅读器
- 读取或修改存储在被动标签中的数据
- 典型应用：手机读取智能海报（Smart Poster）

**2. 卡模拟模式（Card Emulation Mode，PICC，ISO 14443）**
- 设备表现为一个被动目标（PICC）
- 与外部主动阅读器交互
- 典型应用：手机作为交通票、Google Wallet 移动支付

**3. 点对点模式（Peer-to-Peer Mode，ISO 18092）**
- 发起方和目标方都发射功率
- 建立双向数据连接
- 典型应用：两台手机交换虚拟名片、Android Beam

> [!important]+ 模式与标准的对应关系
>
> 三种模式对应不同的 ISO 标准，这是理解 NFC 安全分析的基础：
> - 读写模式与卡模拟模式都基于 **ISO 14443**（邻近卡标准）
> - 点对点模式基于 **ISO 18092**（NFCIP-1）
>
> 不同模式的安全特性不同，例如卡模拟模式涉及支付凭证保护，而点对点模式更关注双向通信的机密性。

### 1.3 NFC 数据交换原理：PCD 与 PICC (page 23)

NFC 数据交换基于两个核心角色，它们源自 ISO 14443 标准：

- **PCD（Proximity Coupling Device，邻近耦合设备）**：即主动设备（阅读器）
  - 连接电源
  - 产生电磁场用于数据交换和为被动设备供电

- **PICC（Proximity Integrated Circuit Card，邻近集成电路卡）**：即被动设备（NFC 标签）
  - 从主动设备（PCD）**收集能量（Harvest power）**
  - 通过负载调制等方式回传数据

> [!note]+ 能量收集是 NFC 的关键
>
> 被动 NFC 标签没有自带电源，它完全依赖 PCD 产生的 13.56 MHz 电磁场感应供电。这也是 NFC 通信距离被限制在几厘米内的根本原因——只有在这个近距离范围内，电磁场强度才足以驱动标签芯片工作。这种"近距离"特性既是 NFC 的局限，也是其物理安全优势（难以远距离窃听）。

### 1.4 阅读器与标签交互流程 (page 7)

NFC 阅读器（PCD）与标签（PICC）之间的标准交互流程如下：

```
PCD (阅读器)                          PICC (标签)
    |                                      |
    |------- REQA (Request Type A) ------->|   请求
    |<----- ATQA (Answer To Request) ------|   应答
    |                                      |
    |======= Anti-collision loop =========|   防冲突循环
    |<----- UID + SAK ---------------------|   返回 UID 和 SAK
    |                                      |
    |------- RATS (Request for ATS) ------>|   请求选择应答
    |<----- ATS (Answer To Select) --------|   应答选择
    |                                      |
    |------- Command APDU ---------------->|   命令
    |<----- Response APDU -----------------|   响应
    |              ...                     |
```

各阶段含义：

1. **REQA / ATQA**：阅读器发送请求，标签应答，建立初始联系
2. **防冲突循环（Anti-collision loop）**：当多张卡同时进入磁场时，逐一识别
3. **UID + SAK**：返回唯一标识符（UID）和选择确认（SAK）
4. **RATS / ATS**：请求并获取"选择应答"，进入更高层协议
5. **APDU 交换**：应用协议数据单元（APDU）的命令-响应交互

> [!definition]+ 关键术语
>
> - **UID（Unique Identifier）**：标签的唯一标识符
> - **SAK（Select Acknowledge）**：选择确认，标识标签类型
> - **APDU（Application Protocol Data Unit）**：应用层协议数据单元，是智能卡通信的基本单位

## 2. NFC Forum 标签类型 (page 44)

NFC Forum 定义了四种标准标签类型，它们在存储结构、协议、容量和安全性上各有差异。理解这些差异对于选择合适的标签和安全方案至关重要。

| 属性 | Type 1 | Type 2 | Type 3 | Type 4 |
|------|--------|--------|--------|--------|
| **传输协议** | ISO 14443A | ISO 14443A | ISO 18092 | ISO 14443A |
| **UID 长度** | 4 或 7 字节 | 4 或 7 字节 | 8 字节 | 7 字节 |
| **内存大小** | 96 字节（最大 2 KB） | 64 字节（最大 2 KB） | 可变（最大 1 MB） | 可变（最大 32 KB） |
| **内存组织** | 12 块，每块 8 字节 | 16 页，每页 4 字节 | 块，每块 16 字节 | 基于智能卡 |
| **OTP 位** | 48 位 | 32 位 | — | — |
| **锁定位** | 16 位 | 16/32 位 | — | — |
| **可重写** | 直到锁定 | 直到锁定 | 预定义 | 发行方定义 |
| **冲突保护** | 否 | 是 | 是 | 是 |
| **传输速率** | 106 kbit/s | 106 kbit/s | 212 或 424 kbit/s | 106/212/424 kbit/s |
| **典型示例** | Topaz | Ultralight | FeliCa Lite | Java Card、DESFire |

> [!summary]+ 标签类型选择要点
>
> - **Type 1 / Type 2**：低成本、小容量存储标签，适合简单识别场景
> - **Type 3**：基于 Sony FeliCa 技术，容量大、速率高，日本交通卡广泛使用
> - **Type 4**：基于智能卡，支持复杂应用和访问控制，适合支付等高安全场景
>
> 标签类型直接决定了可用的安全机制：只有 Type 4 这类智能卡标签才能运行完整的加密认证协议。

## 3. NFC 安全机制与标签分级

### 3.1 NFC 协议本身的安全空白 (page 3)

这是理解 NFC 安全的核心出发点：**NFC 传输协议本身不定义任何特定的加密或安全机制**。

具体而言：

- **ISO 14443**：覆盖读写模式和卡模拟模式，但不包含加密
- **ISO 18092**：覆盖点对点模式，同样不包含加密
- **NDEF 规范**：定义了用于完整性保护的**签名（Signature）方案**，但存在严重局限

> [!attention]+ NDEF Signature 的局限
>
> NDEF 签名方案虽然提供完整性保护，但**无法防止内容克隆**，因为：
> - 签名**不覆盖卡的 UID**，攻击者可以将签名复制到另一张卡上
> - **不包含阅读器认证**用于访问控制
>
> 因此，**密码学安全必须由 NFC 应用自行定义**。NFC 规范本身不提供任何密码学安全保障。

### 3.2 带有安全功能的存储标签 (page 2, 4)

虽然 NFC 协议本身不安全，但部分存储标签内置了安全功能，形成了一个安全等级递进的标签家族：

**MIFARE Ultralight（基础型）**
- 仅包含 UID、锁定位（lock bytes）、OTP
- 无加密认证

**MIFARE Ultralight C（增强型）(page 2)**
- 内存组织与 Ultralight 类似
- 内存更大：**192 字节**
- 额外的 **32 位单向计数器（one way counter）**
- 使用**认证密钥（authentication key）**进行访问控制
- 支持写入保护或读写双向保护
- 采用 **Triple-DES（3DES）认证**

**MIFARE DESFire (page 4)**
- 支持 **Triple-DES / AES 双向认证（mutual authentication）**
- 带有**文件系统和访问控制列表（ACL）**
- 安全性最高

> [!note]+ 安全等级递进
>
> 从 Ultralight → Ultralight C → DESFire，标签的安全能力逐步增强：
> - Ultralight：仅物理锁定，无密码学
> - Ultralight C：引入 3DES 认证和单向计数器（防重放）
> - DESFire：完整文件系统 + AES/3DES 双向认证 + ACL
>
> 单向计数器（one way counter）是 Ultralight C 的关键防重放机制——它只能递增不能回退，适合实现"一次性使用"标签（如票务）。

### 3.3 非接触式智能卡 (page 4)

在安全等级的最顶端是**非接触式智能卡（Contactless Smart Cards）**，它们本质上是带有射频接口的完整计算机：

- **CPU 和操作系统**：具备完整处理能力
- **防篡改处理环境（Tamper-resistant processing environment）**：物理防护
- **安全密码处理器（Secure crypto-processor）**：专用加密硬件
- **安全文件系统（Secure file system）**：受保护的存储

典型示例：
- **JavaCard**：可运行 Java Applet 的智能卡平台
- **EMV 非接触式借记/贷记卡**：银行支付卡标准

> [!important]+ 存储卡与智能卡的界限
>
> 课件指出："存储卡（memory cards）和智能卡（smart cards）之间的区别并不总是泾渭分明的。"
>
> 实际上，随着标签内置安全功能的增强（如 Ultralight C 的 3DES），低端存储标签与高端智能卡之间的界限正在模糊化。但核心区别在于：**智能卡具备完整的可编程 CPU 和操作系统**，能够运行任意复杂的认证协议和应用逻辑。

## 4. NDEF 数据交换格式

### 4.1 NDEF 消息结构 (page 22)

**NDEF（NFC Data Exchange Format）** 是 NFC 的消息封装格式，用于在以下两者之间交换数据：

- 两台 NFC 设备之间，或
- 一台 NFC 设备与一个标签之间

NDEF 消息的结构如下：

```
+--------------------------------------------------+
|                  NDEF Message                    |
+----------+--------+--------+---------------------+
|  NDEF    |  NDEF  |  NDEF  |      ...           |
|  Record  |  Record|  Record|                    |
|    1     |    2   |    3   |                    |
+----------+--------+--------+---------------------+
     |        |        |
     v        v        v
+-------+--+
|Header |Payload|
+-------+--+
```

每条 NDEF 消息包含**一条或多条 NDEF 记录（NDEF Record）**，每条记录由两部分组成：

- **头部（Header）**：定义有效载荷（Payload）的属性
  - NDEF 记录的起始和结束标志
  - **记录类型定义（RTD, Record Type Definition）**：有效载荷的数据类型
  - 有效载荷的长度等
- **有效载荷（Payload）**：实际的 NFC 应用数据

> [!definition]+ NDEF 的定位
>
> NDEF 是一个**封装格式（encapsulation format）**，而非安全协议。它解决的是"如何把多种类型的应用数据打包并通过 NFC 传输"的问题，但**不解决数据如何被保护**的问题。这正是为什么需要额外的 Signature RTD 来补充完整性保护。

### 4.2 Signature RTD 与完整性保护 (page 24)

**Signature RTD（Signature Record Type Definition）** 为 NDEF 消息提供**完整性和真实性（integrity and authenticity）**保护。

**Signature RTD 包含的内容：**

- **签名（Signature）**，支持两种算法：
  - **RSA（1024 位）** 配合 SHA-1 和 PKCS#1 v1.5 填充或 PSS
  - **ECDSA（P-192）** 配合 SHA-1，无填充
- **证书链（Certificate chain）**
- 或者，指向签名的**引用位置（reference location）**

**签名的覆盖范围规则：**

Signature 记录的签名覆盖范围有两种情况：
1. 覆盖**所有前序记录**（从记录 1 开始），或
2. 仅覆盖**位于前一个 Signature 记录之后的那条记录**

```
NDEF Record 1 ─┐
NDEF Record 2 ─┤  签名覆盖 Record 1 & 2
                │
Signature Record 1
                │
NDEF Record 3 ─┘  仅签名 Record 3
                │
Signature Record 2
```

> [!attention]+ Signature RTD 的克隆漏洞
>
> 课件明确指出 Signature RTD 存在**克隆（Cloning）漏洞**：攻击者可以用另一张"有效"的标签替换原标签。
>
> 根本原因在于：**签名不覆盖标签的 UID**。因此，攻击者可以：
> 1. 复制一个合法标签上的 NDEF 消息和签名到另一张标签
> 2. 由于签名验证时不检查 UID，新标签也能通过验证
>
> 这使得 Signature RTD 无法提供真正的防克隆保护，只能保证"内容未被篡改"，但无法保证"内容来自同一张物理标签"。

## 5. 帧等待时间（FWT）

### 5.1 FWT 的定义与作用 (page 6)

**帧等待时间（Frame Waiting Time, FWT）** 是 NFC/ISO 14443 协议中的一个关键时间参数，用于约束标签（PICC）对阅读器（PCD）命令的响应时间。

```
时间轴 ──────────────────────────────────────────────>

PCD: |==== Frame sent by PCD ====|
                                  ^
                                  | t < FWT
                                  v
PICC:                             |==== Frame sent by PICC ====|
```

**核心规则**：标签必须在 **FWT 时间窗口内**完成对阅读器命令的响应。即从 PCD 发送完一帧到 PICC 开始回送响应帧，间隔必须满足 $t < \text{FWT}$。

> [!note]+ FWT 的安全意义
>
> FWT 不仅仅是一个性能参数，它还是**中继攻击（Relay Attack）防御**的关键时间约束。中继攻击需要通过网络转发 APDU，会引入额外延迟；如果 FWT 设置得足够短，理论上可以检测并拒绝超时的中继事务。这正是后续中继攻击对策的理论基础。

### 5.2 ATS 帧结构与 FWI 参数 (page 8)

FWT 的具体值由标签在 **ATS（Answer To Select）** 帧中通过 **FWI（Frame Waiting Time Integer）** 参数通告。ATS 帧的完整结构如下：

| 字段 | 含义 |
|------|------|
| TL | 长度字节（Length Byte） |
| T0 | 格式字节（Format Byte） |
| TA, TB, TC | 接口字节（Interface Bytes） |
| T1...Tk | 历史字节（Historical Bytes，ISO/IEC 7816-4） |
| CRC 1, CRC 2 | 循环冗余校验 |

其中 **TB 字节**携带两个关键时间参数：

- **FWI（Frame Waiting Time Integer）**：帧等待时间整数
- **SFGI（Start-up Frame Guard Time）**：启动帧保护时间整数

**FWT 的计算公式：**

$$
\text{FWT} = \frac{256 \times 16}{f_c} \times 2^{\text{FWI}}
$$

其中 $f_c = 13.56 \times 10^6$ Hz 为载波频率。

**典型 FWI 值对应的 FWT：**

| FWI | 计算 | FWT 值 |
|-----|------|--------|
| 0（最小） | $\frac{256 \times 16}{13.56 \times 10^6} \times 2^0$ | ≈ 303 μs |
| 4 | $\frac{256 \times 16}{13.56 \times 10^6} \times 2^4$ | ≈ 4833 μs（约 4.8 ms） |
| 8 | $\frac{256 \times 16}{13.56 \times 10^6} \times 2^8$ | ≈ 77 ms |
| 14（最大） | $\frac{256 \times 16}{13.56 \times 10^6} \times 2^{14}$ | ≈ 4949 ms（约 4.9 s） |

> [!example]+ FWT 计算示例
>
> 以 MIFARE DESFire 为例，其默认 FWI = 0x8（即 8），因此：
>
> $$\text{FWT} = \frac{256 \times 16}{13.56 \times 10^6} \times 2^8 \approx 77 \text{ ms}$$
>
> 这意味着 DESFire 标签承诺在 77 ms 内响应阅读器的命令。任何超过这个时间的响应都应被视为异常。

### 5.3 FWT 实现缺陷 (page 9)

课件通过实测揭示了一个严重的安全相关实现缺陷：

**观察结果（基于 Android Jelly Bean 4.1.2）：**

- MIFARE DESFire 默认 FWI = 0x8，对应 FWT = 77 ms
- 但 **Nexus S 手机的响应时间远超 77 ms，达到约 430 ms**
- **修改 FWI 参数并不影响实际响应时间**
- 研究者假设这是**固定的 FWT 实现（fixed FWT implementation）**
- **阅读器经常忽略 FWT 配置（readers often ignore FWT configurations）**

> [!attention]+ FWT 实现缺陷的安全后果
>
> 这一缺陷直接削弱了 FWT 作为中继攻击防御手段的有效性：
>
> 1. **阅读器不严格执行 FWT**：即使标签通告了较短的 FWT，阅读器仍可能容忍远超该时间的响应
> 2. **手机响应时间远大于规范**：430 ms vs 77 ms，给中继攻击留下了巨大的时间窗口
> 3. **参数可配置性失效**：修改 FWI 无法真正缩短响应时间
>
> 这意味着**单纯依赖 FWT 来防御中继攻击在实践中是不可靠的**，需要结合其他机制（如位置验证、应用层超时）。

## 6. NFC 中继攻击

### 6.1 中继攻击原理 (page 5, 37)

**中继攻击（Relay Attack）** 是 NFC 安全中最具代表性的攻击形式，本质上是一种中间人（Man-in-the-Middle, MITM）攻击。

**经典场景（page 5）：** 将你口袋里的非接触式 EMV 支付凭证**中继到远处的商店**完成交易。

**攻击架构（page 37）：**

```
   [受害者的手机]              [攻击者的设备]              [POS 终端]
   (在受害者口袋里)            (在远处商店)
        |                            |                          |
        |<== "Mole" 阅读器 ==========|                          |
        |   (靠近受害者)             |                          |
        |                            |======= 攻击者手机 ======>|
        |                            |   (靠近 POS)             |
        |                            |                          |
        |<======= APDU 通过 TCP/IP 转发 =======================>|
```

**攻击的两个代理角色：**

1. **"鼹鼠"（Mole）阅读器**：靠近目标移动设备，读取其 NFC 通信
2. **攻击者的手机**：靠近 POS 终端，模拟支付卡
3. 两者之间通过 **TCP/IP 网络**转发 APDU

> [!note]+ 中继攻击的关键特点
>
> 课件（page 5）强调中继攻击的两个技术要点：
>
> - **需要在代理令牌上实现卡模拟（card emulation）**：攻击者设备必须能模拟成一张支付卡
> - **不需要 UID 欺骗（UID spoofing）**：因为 **EMV 标准不使用 UID** 进行交易认证
>
> 这意味着攻击者无需破解 UID 即可实施中继——EMV 的安全模型假设物理邻近性，而中继攻击正是打破这一假设。

### 6.2 通过恶意 App 的中继 (page 39)

除了物理代理，中继攻击还可以通过**恶意应用程序**实施：

- 针对 **Google Wallet** 有效
- 原因：Google Wallet **将支付凭证暴露给接触式接口（contact interface）**
- 前提条件：需要 **root 权限**来绕过 SE API 的签名认证

> [!important]+ 接触式接口暴露的风险
>
> 正常情况下，支付凭证应仅通过非接触式（NFC）接口访问。Google Wallet 当时的实现将其暴露在接触式接口上，意味着设备上的其他应用（在获得 root 后）可以读取这些凭证，从而绕过 NFC 的物理邻近性假设，实现"软件中继"。

### 6.3 中继攻击的局限与对策 (page 38, 40, 41)

**攻击的局限（page 38）：**

- 目标手机的支付应用必须**已解锁**
- 例如 Google Wallet 要求输入 PIN 才能解锁

**对策（page 40）：**

1. **非接触式 POS 终端应对所有事务强制超时（time-out）**
   - 原理：中继攻击需要网络通信，会引入延迟
   - 局限：**实践中不太可行**，因为合法事务也可能因错误而产生延迟

2. **使用位置信息标记可疑事务**
   - 检测目标手机是否真的在 POS 附近
   - 局限：**需要目标 GPS 处于激活状态**，且需要**用户同意**

**Google Wallet 的修复（page 41）：**

- Google Wallet **不再容易受到第二种攻击（恶意 App 中继）的影响**
- 修复方式：**不再通过接触式接口暴露支付 applet**

> [!summary]+ 中继攻击防御的困境
>
> 中继攻击的防御面临根本性困境：
>
> | 对策 | 有效性 | 实践障碍 |
> |------|--------|----------|
> | FWT 超时 | 理论有效 | 阅读器不严格执行、手机响应慢 |
> | POS 事务超时 | 部分有效 | 合法事务也可能超时，误报率高 |
> | 位置验证 | 概念有效 | 需 GPS 激活和用户同意 |
> | 接口隔离 | 有效 | 需要正确实现，Google Wallet 曾失误 |
>
> 核心矛盾在于：NFC 的安全模型依赖**物理邻近性**作为隐式认证因素，而中继攻击正是通过技术手段"延伸"了这种邻近性。任何纯粹基于时间的防御都难以区分"中继延迟"和"正常处理延迟"。

## 7. 手机上的 NFC 与安全威胁

### 7.1 NFC 在手机上的集成 (page 10, 11)

**支持 NFC 的手机（page 10）：**

NFC 在手机中的集成正在快速增长，典型机型包括：

- Nokia Lumia 610 及以上
- Nexus S 及以上
- Samsung Galaxy S II plus 及以上
- BlackBerry 10
- iPhone 6 及以上
- 以及更多机型

**手机 NFC 的功能支持（page 11）：**

- 主要支持**读写（Read/Write）**和**点对点（P2P）**模式
- 部分手机平台包含**安全元件（Secure Element, SE）**，这是实现**卡模拟（Card Emulation）**所必需的
- **主机卡模拟 API（Host Card Emulation API, HCE）** 在 Android 4.4 及以上版本可用

**当前典型应用：**

- NDEF 标签读写
- FourSquare 签到（check-ins）
- Samsung TecTiles
- 公共交通票务
- 移动支付（Mobile payment）
- 忠诚卡（Loyalty card）

> [!note]+ 安全元件（SE）与 HCE 的演进
>
> 卡模拟模式需要安全元件来安全存储支付凭证。早期依赖硬件 SE（如嵌入式 SE、SIM 卡、MicroSD）。Android 4.4 引入的 **HCE（Host Card Emulation）** 允许通过软件实现卡模拟，无需硬件 SE，这降低了部署门槛但也带来了新的安全考量（凭证存储在主机侧，需要 TEE 保护）。

### 7.2 手机 NFC 面临的威胁 (page 13)

手机 NFC 面临多种威胁，主要分为以下几类：

**1. 拒绝服务攻击（Denial of Service）**

- 手机 NFC 协议栈会对进入其 NFC 范围内的**任何标签**做出反应
- 某些**格式错误的标签（mal-formatted tags）**会导致协议栈阻塞（jam the stack）
- 此外，安全元件（SE）中的大多数卡管理器在**连续 10 次认证失败后会自我锁定**

**2. 通过 NFC 投递恶意软件（Malware delivery via NFC）**

移动操作系统会读取 NDEF 消息并打开对应的应用程序：

- 包含 URL 的 NDEF 会让手机在默认浏览器中打开该 URL
  - 攻击：NDEF 中的 URL 指向**恶意软件下载页面**
- 包含恶意内容的 NFC 消息
- 通过 NFC 发送**恶意文件**以利用 Android 文档查看器漏洞
- 通过 NFC 执行 **USSD（Unstructured Supplementary Service Data）** 码

> [!attention]+ NFC 作为攻击载体的特殊性
>
> NFC 的"自动打开应用"特性使其成为一个高效的攻击载体：
>
> - **无需用户交互**：手机靠近恶意标签即可触发
> - **隐蔽性强**：用户可能不知道靠近标签就执行了操作
> - **攻击面广**：浏览器、文档查看器、USSD 处理等都可能成为目标
>
> 这与传统的"用户主动点击链接"攻击模型不同，NFC 攻击利用的是物理邻近性触发的自动化处理流程。

> [!important]+ 手机 NFC 安全的依赖关系
>
> 课件（page 14）指出，手机上基于 NFC 的金融应用安全**依赖于手机平台提供的安全机制**：
>
> - **沙箱（Sandboxes）**
> - **基于权限的访问控制（Permission based access control）**
>
> 这些机制应当防御：
> - **移动恶意软件（mobile malwares）**
> - **手机用户的恶意意图（Ill intent of the phone user）**，例如用户可能获取 root 权限来修改票务价值
>
> 更进一步，需要**即使操作系统被攻破（OS is compromised）也能保证安全**——这正是可信执行环境（TEE）存在的意义。

## 8. 移动支付安全：Google Wallet 案例分析

Google Wallet 是移动支付安全的经典案例，课件（page 30-41）详细分析了其漏洞。

### 8.1 PIN 存储漏洞 (page 31-33)

**漏洞背景（page 31）：**

- Google Wallet 交易需要**输入 PIN**
- 仅允许**6 次尝试**
- 但是，**攻击者偷走设备并 root 后，可以从加盐哈希中提取 PIN**
- 原因：**PIN 没有存储在安全元件（SE）上**
- 将 PIN 存储在 SE 上会使银行对因 PIN 被盗导致的泄露**承担责任**

**PIN 存储机制（page 32）：**

PIN 通过以下方式存储：

- 使用 **64 位随机值作为盐（salt）**
- 用 **SHA-256 进行一轮（one round）哈希**

**哈希存储位置（page 33）：**

- 盐和哈希存储在 Google Wallet 的 `/data` 目录下的 **SQLite 数据库**中：
  ```
  /data/data/com.google.android.apps.walletnfcrel/databases/walletDatastore
  ```
- **"Wallet Cracker"** 工具只需**尝试所有 10000 个四位数字 PIN** 即可从哈希中恢复 PIN

> [!example]+ 漏洞的根本原因分析
>
> 这个漏洞是多个设计缺陷叠加的结果：
>
> | 缺陷 | 说明 |
> |------|------|
> | **PIN 不在 SE 上** | 存储在普通文件系统中，root 后可读 |
> | **仅一轮 SHA-256** | 没有使用 PBKDF2 等慢哈希，暴力破解极快 |
> | **4 位 PIN** | 仅 10000 种可能，暴力破解 trivial |
> | **64 位盐无效** | 盐虽防彩虹表，但无法防针对单 PIN 的暴力破解 |
>
> 正确的做法是：将 PIN 及其尝试计数器都存储在 SE 上，由 SE 执行验证。课件指出，Google 不这样做的原因是**责任归属问题**——一旦 PIN 在 SE 上，银行就要为 PIN 泄露负责。

### 8.2 Google 的回应与对策 (page 35, 36)

**Google 的官方回应（page 35）：**

- **不要在已 root 的手机上运行 Google Wallet**
- 但这**并不令人安心**，因为**小偷可以自行 root 你的手机**
- 更好的做法是：**在 SE 上执行 PIN 存储和验证**
- 同时**将 PIN 尝试计数器也存储在 SE 上**

**针对 Google Wallet Cracker 的对策（page 36）：**

1. 不要 root 你的设备
2. 启用 Android 锁屏
3. 禁用 ADB 调试
4. 保持补丁更新

> [!note]+ 对策的局限性
>
> 课件列出的用户侧对策（不 root、锁屏、禁用 ADB、打补丁）都是**防御性建议**，但它们假设用户主动配合。而真正的攻击场景（设备被盗后被 root）下，这些对策全部失效。**根本性的修复必须从架构层面入手**：将敏感凭证和验证逻辑移入硬件隔离的 SE 或 TEE。

## 9. 可信执行环境与安全元件

### 9.1 手机上的安全执行要求 (page 15, 16)

为了在操作系统可能被攻破的情况下仍保证 NFC 应用的安全，手机需要一套**安全执行（Secure Execution）**能力。课件（page 15-16）定义了这些核心要求：

**基础安全执行能力（page 15）：**

1. **隔离执行（Isolate Execution）**
   - 将安全敏感代码在**完全隔离**的环境中执行
   - 确保应用数据的**完整性（integrity）**和**运行时机密性（run-time secrecy）**

2. **安全存储（Secure Storage）**
   - 保护存储数据免受未授权访问
   - 用于存储密码、密钥、凭证等

3. **远程证明（Remote Attestation）**
   - 在交互前**远程验证**特定应用的真实性
   - 信任根度量（Root of trust measurement）
   - **动态信任根度量（Dynamic Root of Trust Measurement, DRTM）**

**扩展安全执行能力（page 16）：**

4. **安全配置（Secure Provisioning）**
   - 从远程服务器**通过空中（over the air）**安全部署应用模块到特定用户设备
   - 支持应用在设备间的**迁移（migration）**

5. **可信路径（Trusted Path）**
   - 确保端点之间**未篡改的通信通道**
   - 例如：NFC 前端控制器与隔离执行环境之间的**直接物理连接**

> [!definition]+ DRTM（动态信任根度量）
>
> 传统的静态信任根度量（如 TPM 的 PCR）在启动时建立信任链，但一旦系统运行就无法更新。**DRTM（Dynamic Root of Trust Measurement）** 允许在系统运行期间**动态地**为特定安全敏感代码建立新的信任根，无需重启。这使得即使主操作系统被攻破，也能在受保护环境中安全执行关键操作（如支付验证）。

### 9.2 可用的安全元件类型 (page 17)

课件列举了手机上可用的多种安全元件（Secure Element, SE）形式：

| 安全元件类型 | 特点 | 典型案例 |
|--------------|------|----------|
| **非接触式贴纸（Contactless stickers）** | 独立于手机操作系统 | Elisa Lompakko |
| **UICC（通用集成电路卡）** | 即 SIM 卡，移动运营商首选 | Orange Quick Tap |
| **安全 MicroSD** | 由部分银行使用 | 台湾部分银行 |
| **嵌入式安全元件（Embedded SE）** | 焊接在主板上 | Google Wallet |
| **可编程 TEE（Programmable TEE）** | 软件隔离执行环境 | ObC（On-board Credential）、Mobicore |

> [!note]+ SE 形式的权衡
>
> 不同 SE 形式反映了不同利益相关方的博弈：
>
> - **运营商（MNO）偏好 UICC/SIM**：因为 SIM 卡由他们控制
> - **银行偏好嵌入式 SE 或 MicroSD**：避免依赖运营商
> - **贴纸方案最简单**：完全独立于手机 OS，但功能有限
> - **可编程 TEE**：无需独立硬件，但安全性依赖 CPU 的隔离扩展（如 ARM TrustZone）

### 9.3 安全元件架构：以 SIM 为例 (page 18)

SIM 卡本质上是一张**多应用智能卡（multi-application smart card）**，其架构如下：

```
+---------------------------------------------------+
|              Card Manager（卡管理器）              |
+---------------------------------------------------+
|         |              |              |           |
| 信用卡   | 公交票务      | 发行方安全    | 第三方    |
|  Applet |  Applet      |  域（ISD）    | 安全域    |
|         |              |              |           |
+---------+--------------+--------------+-----------+
|  应用防火墙  |  应用防火墙（Application Firewall）   |
+---------------------------------------------------+
```

**关键设计：**

- 每个服务提供商在卡上创建一个**独立的安全域（Security Domain）**
- 不同 applet 之间通过**应用防火墙（Application Firewall）**隔离
- 包含**发行方安全域（Issuer Security Domain）**和**第三方安全域**

**存在的问题：**

- 增加了**卡管理器（card manager）的复杂性**
- **空中（over-the-air）安装新 applet 具有挑战性**

### 9.4 TSM 与 HCE (page 19, 20)

**可信服务管理器（Trusted Service Manager, TSM）(page 19)：**

由于服务提供商（银行、公交等）和移动运营商（MNO）之间需要信任中介，引入了 TSM 模型：

```
   服务提供商                  TSM                  移动运营商（MNO）
  (银行、公交等)            (单一可信实体)              |
       |                       |                       |
       |                       v                       v
       |                  +----------------+      +----------+
       |                  | 安全地分发并   |----->| SIM /    |
       |                  | 个性化 SP 应用 |      | 嵌入式SE |
       |                  | (OTA 个性化)   |      | / μSD    |
       |                  +----------------+      +----------+
       |                       |
       |                       +-- 验证用户设备和 SE 能力与资源
       |                       +-- 管理应用生命周期
```

**TSM 的核心职责：**

- 作为**单一可信实体**同时服务 MNO 和服务提供商
- **通过空中（OTA）安全分发并个性化**服务提供商应用到客户的 SE
- **验证**用户设备和 SE 的**能力与资源**
- **管理应用的生命周期**

**主机卡模拟（Host Card Emulation, HCE）(page 20)：**

HCE 提供了两种卡模拟架构的对比：

```
架构一：基于安全元件的 NFC 卡模拟
NFC Reader <--> NFC Controller <--> [Secure Element]

架构二：基于可编程 TEE 的 NFC 卡模拟
NFC Reader <--> NFC Controller <--> Host CPU <--> [Programmable TEE]
```

> [!important]+ HCE 的革命性意义
>
> HCE（Android 4.4+）使 NFC 卡模拟**不再依赖硬件安全元件**：
>
> - **传统架构**：NFC Controller 直连 Secure Element，凭证存储在 SE 中
> - **HCE 架构**：通过 Host CPU 上的应用（配合 TEE）实现卡模拟
>
> 这降低了移动支付的部署门槛（无需与运营商协商 SE 访问权限），但也要求 TEE 提供等效的安全保证，否则会重蹈 Google Wallet PIN 存储漏洞的覆辙。

## 10. NFC 应用场景

### 10.1 基于 UID 的访问控制 (page 25)

最简单的 NFC 应用直接利用标签的 **UID** 作为凭证：

**工作原理：**

- 基于**标签 UID** 的简单访问控制应用
- NFC 标签用作识别用户的**凭证（credential）**
- 阅读器必须连接到**后端数据库**
- 后端服务器维护**访问策略（access policies）**

**优点：**

- **简单且廉价**的解决方案
- 常见假设："**UID 不容易被伪造！！**"
- 适合**小规模商业**场景

**缺点：**

- 后端复杂性随**客户数量增加**而上升
- 阅读器需要**始终连接**到后端服务器

> [!attention]+ "UID 不易伪造"假设的危险性
>
> 课件引用的"UID cannot be faked easily"是一个**有条件的假设**。实际上：
>
> - 许多标签的 UID 是**可写的**（特别是可编程卡）
> - 即使是"只读"UID，也可以用**UID 可写卡（如 Magic UID 卡）**克隆
> - 这正是中继攻击和克隆攻击能够成功的基础
>
> 因此，**仅依赖 UID 的访问控制是不安全的**，应结合加密认证。

### 10.2 一次性/有限次使用标签（票务）(page 26)

**事件票务（Event Ticketing）** 场景使用 MIFARE Ultralight / Ultralight C 标签实现一次性或有限次使用：

**实现机制：**

- 阅读器实现**密码学功能**
- **密钥分散（Key diversification）**：例如 $\text{Hash}(\text{UID} + \text{Master key})$
- **加密数据并存储**密文在标签上
- 从标签**读取密文并解密**
- 使用 **OTP 字节作为递增计数器（incremental counter）**
- 使用**锁定位（Lock bytes）防止重写**
- **MAC** 用于完整性保护
- **认证密钥**用于访问控制

> [!example]+ 票务防重用的设计
>
> 一次性票务的核心是**防重放**和**防篡改**：
>
> 1. **OTP 计数器**：每次检票递增，由于是"单向"的，无法回退——已使用的票无法"重置"
> 2. **锁定位**：一旦设置，对应存储区永久不可修改，防止攻击者篡改票务数据
> 3. **密钥分散**：每张卡的密钥基于其 UID 派生，即使一张卡被破解也不会泄露其他卡的密钥
> 4. **MAC**：防止数据被篡改
>
> 这种设计在标签资源极其有限的情况下，巧妙地利用了硬件特性（OTP、锁定位）配合轻量级密码学实现了可接受的安全性。

### 10.3 公共交通应用 (page 27)

公共交通是 NFC 最成熟的应用领域之一：

**广泛使用的专有解决方案：**

- **MIFARE Classic**
- **MIFARE Ultralight / Ultralight C**
- **MIFARE DESFire EV1**

**技术特点：**

- 使用**对称密码学（Symmetric crypto）**：Triple-DES、AES
- **价值（Value）存储在卡上**

**标准：**

- **ITSO**：使用非接触式智能客户媒体的可互操作公共交通票务标准

> [!note]+ 公交票务的安全挑战
>
> 公交系统需要在**高吞吐量**（每秒处理大量乘客）和**安全性**之间平衡。MIFARE Classic 曾因 Crypto-1 流密码的破解而暴露严重漏洞，这也是行业向 DESFire（AES）迁移的原因。价值存储在卡上意味着离线交易，这要求卡本身具备强认证能力。

### 10.4 开放支付票务 (page 28, 29)

**开放支付票务（Open Payment Ticketing）** 是一种更现代的架构，将票务与支付解耦：

**Smart Card Alliance 提案（page 28）：**

- 每位旅客在由**服务提供商（Service Provider, SP）**运营的服务器云中拥有一个**旅行账户（travel account）**
- 交通卡**只存储用户身份和凭证**
- 用户身份在车站闸机的阅读器处验证
- 票务身份和行程信息发送到**后端服务器**
- 后端服务器计算票价并将信息转发给 SP 进行**收款**
- SP **收款**

**优势：**

- 允许使用**来自不同 SP 的凭证**
- 例如：银行卡、SIM 卡、国民身份证等

**移动手机上的开放支付票务（page 29）：**

涉及三方角色：

| 角色 | 职责 |
|------|------|
| **交通管理部门（Transport Authority）** | 运营交通系统；基于 ID 和行程距离计算票价；收集票务证据用于审计 |
| **服务提供商（如银行或移动运营商）** | 管理客户关系和旅行账户；签发旅行凭证；直接从手机和交通部门收集证据用于审计；向客户收款（预付或信用） |
| ** fare 计算服务** | 票价计算 |

> [!summary]+ NFC 应用场景的安全等级演进
>
> 从 UID 访问控制到开放支付票务，NFC 应用的安全架构逐步成熟：
>
> | 应用场景 | 安全机制 | 凭证存储位置 | 在线/离线 |
> |----------|----------|--------------|-----------|
> | UID 访问控制 | 仅 UID | 标签 UID | 在线（依赖后端） |
> | 事件票务 | OTP + 锁定位 + MAC + 密钥分散 | 标签（密文） | 离线为主 |
> | 公交票务 | 对称加密（3DES/AES） | 卡上（价值） | 离线 |
> | 开放支付票务 | 账户制 + 多 SP 凭证 | 云端账户 | 在线 |
>
> 趋势是：**将价值存储从卡端转移到云端账户**，卡仅作为身份凭证。这降低了卡端被攻击的损失，但增加了对后端安全和网络连接的依赖。

## 总结

本讲系统介绍了 NFC 的技术基础与安全挑战，核心要点如下：

1. **NFC 协议本身不提供密码学安全**：ISO 14443/18092 和 NDEF 都不包含加密，安全必须由应用层定义。

2. **标签安全能力分级**：从无加密的 Ultralight，到 3DES 认证的 Ultralight C，再到支持 AES 和文件系统 ACL 的 DESFire，以及完整的非接触式智能卡。

3. **NDEF Signature 存在克隆漏洞**：签名不覆盖 UID，无法防止内容克隆。

4. **FWT 作为中继防御在实践中失效**：阅读器常忽略 FWT 配置，手机响应时间远超规范。

5. **中继攻击是 NFC 核心威胁**：利用物理邻近性假设的破绽，通过代理转发 APDU 实施远程交易。

6. **Google Wallet PIN 漏洞**：PIN 不在 SE 上、仅一轮 SHA-256、4 位 PIN，导致 root 后可暴力破解。

7. **安全执行环境是根本对策**：隔离执行、安全存储、远程证明、DRTM 等能力，通过 SE（SIM/嵌入式/MicroSD）或 TEE/HCE 实现。

8. **应用场景安全演进**：从 UID 识别到云端账户制，价值存储逐步上云，卡端仅作身份凭证。

> [!note]+ 与 RFID 安全的联系
>
> 本讲（NFC）与上一讲（RFID）紧密相关：NFC 本质上是工作在 13.56 MHz 的高频 RFID 标准。两者的安全挑战相似——都受限于标签资源、都面临克隆和中继威胁。区别在于 NFC 更强调**手机集成**和**支付场景**，因此引入了 SE、TEE、HCE 等移动平台特有的安全机制。
